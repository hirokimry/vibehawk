# 💬 コード内コメント作成基準

> [!IMPORTANT]
> **デフォルトはコメントを書かない**。コードが自己説明的であることを優先する。
> コメントを書くのは **WHY が非自明な時のみ**。
> WHAT（コードを読めば分かる挙動）を繰り返さない。

コード内コメントを書くための基準を定める。

対象種別:

- 🐚 シェルスクリプト（bash / sh）
- 📜 TypeScript / JavaScript
- 🐍 Python
- 💎 Ruby
- 🦀 Rust / 🐹 Go / その他言語
- ⚙️ 設定ファイル（JSON / YAML / TOML 等）

書き方ルール（ドキュメント全般）は `document-writing.md` を参照する。

CEO への対話・Issue / PR / コミットの文面は `communication.md` を参照する。

本ルールは「コード内コメントの書き方」を扱う。

## 🎯 対象範囲

本ルールは frontmatter の `paths` で対象を絞る。

- 対象: シェル / TypeScript / Python 等のソースコード。
- 読込タイミング: 対象ファイル編集時に lazy-load。
- 公式仕様: https://code.claude.com/docs/en/memory#path-specific-rules

### 🚫 対象外（paths がマッチしないため本ルールは読み込まれない）

- `.md` ファイル（ドキュメント本文）。
  - `document-writing.md` / `prompt-writing.md` が適用される。
- Issue / PR / レビュー / Bot 通知の文面。
  - GitHub コメント基準（兄弟ルール）で扱う。
- コミットメッセージ本文。
  - `communication.md` の動作主語ルールで扱う。
- 一時ファイル・キャッシュ（リポジトリ外）。

## 🗣️ 動作主語と WHY（MUST）

コードコメントは **WHY を語る**。

- 主語: 「このコードが何を実現するか」「なぜこの実装か」。
- WHAT（コードを読めば分かる挙動）を繰り返さない。
- 仕様の根拠・トレードオフ・例外・歴史的経緯を残す。

### 🔄 Before / After

| 観点 | 🔴 Before（WHAT 繰り返し・NG） | 🟢 After（WHY を残す・OK） |
|------|----------------------------|--------------------------|
| 🐚 shell | `# x に 1 を加算する` + `x=$((x + 1))` | `# 1-indexed 補正（仕様: Issue #123）` |
| 📜 TS | `// ユーザーを取得` + `const user = ...` | `// SSR で cookie 未確定なので fallback` |
| ⚙️ 設定 | `# port を 8080 に設定` | `# Docker 外部公開ポートと一致させる` |

## ✅ 指針（MUST）

コードコメントを書く前・書いた後に以下 6 項目を必ず通す。

1. ✋ **default to no comments**
   - コードを書く前に「コメント無し」を既定とする。
   - 命名・構造・テスト名で意図を表せないか先に検討する。
   - それでも非自明な WHY が残る時だけ書く。

2. 🧭 **WHY を書く（WHAT を書かない）**
   - 仕様の根拠・トレードオフ・例外・歴史的経緯を書く。
   - コードを読めば分かる挙動を繰り返さない。
   - 「なぜこの順序か」「なぜこの値か」を残す。

3. 📍 **出所をリンクする**
   - WHY の出所（Issue 番号 / 仕様書 / 関連ファイル）を添える。
   - 関連 PR や Issue 番号は `#123` 形式で書く。
   - 関連ファイルへのパスは相対パスで書く。

4. 🔁 **コードと一致させる**
   - 設定ファイルのコメントは設定キーの公式な意味を正確に記述する。
   - コード変更時にコメントも同期する。
   - 誤解を招くコメントはバグと同等に扱う。

5. 🗣️ **言語は日本語で書く**
   - コメントは日本語で書く（変数名・関数名は英語のままでよい）。
   - 仕様書が英語でも、実装コメントは日本語で統一する。
   - 例外: 標準ライブラリ・公式ドキュメントの引用部分は原文のまま。

6. 🚫 **コメントアウトされたコードを残さない**
   - 旧実装は git の履歴に残るため、コード中に残さない。
   - 「いつか戻すかも」は YAGNI 違反。
   - 戻したくなったら git log / git blame で復元する。

## ❌ 禁止パターン

以下 6 項目はコードコメントとして機能しない書き方。

書いた後に必ず通す。

- ❌ **WHAT の繰り返し**
  - 読めば分かる挙動をコメントで繰り返す。
  - 「ユーザーを取得する」+ `const user = getUser()`。

- ❌ **誤解を招くコメント**
  - コード変更後にコメントだけ古いまま残る。
  - 設定キーの意味と説明が食い違う。

- ❌ **コメントアウトされたコード**
  - 旧実装をコメントで残す。
  - 「念のため」「いつか戻すかも」は禁止。

- ❌ **Issue 番号無しの TODO / FIXME**
  - 必ず `TODO(#123): ...` のように Issue 番号を添える。
  - 番号無しの TODO は雪だるま式に増える。

- ❌ **個人名 / 個人視点のコメント**
  - 「太郎が書いた処理」「自分はこう考えた」など。
  - 読者が誰でも理解できる客観表現にする。

- ❌ **AI 由来の冗長コメント**
  - 関数の引数を逐一説明するだけのコメント。
  - 自明な型・自明な処理にコメントを付ける。

## 📚 言語別の節

各言語の文化・慣習に従う。

### 🐚 bash / shell

冒頭 + 関数 + 複雑な式の 3 か所が主戦場。

| 配置 | 役割 | 例 |
|------|------|------|
| ファイル冒頭 | 用途を 1 行で示す | `# 用途: PR の base を決定する（Issue #519）` |
| `set -euo pipefail` 直下 | スクリプト全体の目的 1〜3 行 | 「sub-issue 関係から PR の base を決定する」 |
| 関数の上 | 機能の 1 行サマリ | `# 機能: parent issue 番号を取得する` |
| 複雑な正規表現 / パイプ | WHY を 1 行で添える | `# 1-indexed 補正` |

```bash
#!/usr/bin/env bash
# 用途: PR の base ブランチを sub-issue 関係から決定する（Issue #519）
set -euo pipefail

# 機能: parent issue 番号を取得する（404 は sub-issue でない扱い）
fetch_parent() {
  gh api "/repos/${OWNER}/${REPO}/issues/${NUM}/parent" --jq '.number // empty'
}
```

#### 🚫 bash の禁止パターン

- ❌ `# x に 1 を加算する` 等の WHAT 繰り返し。
- ❌ `sed` / `awk` の式に WHAT 解説を付ける。
  - 参照: `shell.md`。

### 📜 TypeScript / JavaScript

JSDoc は **public API のみ** に書く。

| 種別 | コメント方針 |
|------|------------|
| public API | JSDoc で WHY を残す（引数の型は省く） |
| 内部関数 | 命名で意図を表す（コメントは不要） |
| React コンポーネント | 命名で意図を表す（コメントは最小化） |
| 複雑な実装 | `//` で WHY を 1 行で添える |

```typescript
// SSR で cookie 未確定なので fallback ユーザーで初期化する
// 📍 根拠: Issue #234（hydration mismatch 対策）
const user = ssr ? FALLBACK_USER : useAuth().user;
```

#### 🚫 TypeScript / JavaScript の禁止パターン

- ❌ JSDoc で型を再記述する。
  - TypeScript 型システムで十分。
- ❌ 内部関数に JSDoc を付ける。
  - 命名で意図を表す。

### 🐍 Python

docstring は **public API のみ** に書く（PEP 257 準拠）。

| 種別 | コメント方針 |
|------|------------|
| public 関数 / クラス | docstring の 1 行目は命令形で要約 |
| 内部関数 | 型ヒントで意図を表す（docstring は省く） |
| 複雑なロジック | `# Why: ...` 形式のインラインコメント |

```python
def calculate_score(user: User) -> int:
    """ユーザーのスコアを計算する。

    Why: 新規ユーザーには 100 ポイントのボーナスを加算する
    （仕様: docs/scoring.md）。
    """
    return user.points + (100 if user.is_new else 0)
```

#### 🚫 Python の禁止パターン

- ❌ docstring で引数の型を再記述する。
  - 型ヒントで十分。
- ❌ 内部関数全てに docstring を付ける。

### 💎 Ruby

| 種別 | コメント方針 |
|------|------------|
| public メソッド | YARD で WHY を残す |
| private メソッド | 命名で意図を表す（コメント不要） |
| DSL ブロック | WHY を 1 行で添える |

#### 🚫 Ruby の禁止パターン

- ❌ private メソッド全てに YARD コメント。
- ❌ ActiveRecord の DSL に WHAT を繰り返す。

### 🦀 Rust / 🐹 Go / その他

各言語の公式 doc コメント規約に従う。

| 言語 | doc コメント形式 |
|------|----------------|
| Rust | `///` （doc comment）/ `//!`（module doc） |
| Go | `// FuncName ...` の形式 |

WHY を残し、WHAT を繰り返さない原則は全言語共通。

### ⚙️ 設定ファイル（JSON / YAML / TOML 等）

設定キーの **公式な意味** を正確に記述する。

- 設定キーの効果（何が変わるか）を書く。
- 値の根拠（仕様書 / 関連 Issue）を添える。
- デフォルト値からの差分理由を残す。

#### 🚫 設定ファイルの禁止パターン

- ❌ 設定キーの意味と説明が食い違う。
- ❌ 「Magic number」をコメント無しで放置する。
