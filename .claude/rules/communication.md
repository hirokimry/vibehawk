# 🗣️ コミュニケーション規約

> [!IMPORTANT]
> CEO 向け文面は **動作主語**（〜になった／〜できるようになった）で書く。
> マークダウン整形と状態絵文字で **30 秒スキャン** を成立させる。
> コード参照・関数名・diff だけで喋らない。

CEO 向けの対話応答・Issue / PR 本文・コミットメッセージ・監査レポートを書くための規約を定める。

## 👥 読者像

CEO は **代表取締役社長＝経営者** である。

- 根拠: `docs/ai-organization.md` の基本思想 / `.claude/rules/roles.md` の CEO 定義。
- CEO は「ソフトウェアが何をできるようになったか／壊れたか／変わるか」で判断する。
- エージェントの役目はその判断材料を最短で渡すことにある。

## 前提

本規約は CEO への報告窓口を扱う。

- 読み手は CEO。
- 書く前に **読者像 = CEO** を冒頭で固定する。
- 主語と語尾はソフトウェアの **挙動変化** を主語にする。
- スキャン性を担保するためマークダウンと状態絵文字を最大限活用する。

## 📝 対象文面

本規約は以下の文面に適用する。

- CEO への対話応答（エージェントが直接返すテキスト）
- Issue タイトル / 本文
- PR タイトル / 本文
- コミットメッセージ
- 監査レポート（`/audit-cost` / `/audit-security` など）

### 🚫 対象外（正確性優先で良い）

以下は正確性・網羅性を優先し、本規約の「動作主語」ルールを適用しない。

- `.claude/rules/` のエージェント向けルール（本ファイル含む）
- コード内コメント（`code-comments.md` が管轄）
- `docs/` の技術仕様書（`document-writing.md` が管轄）
- プランファイル（`~/.cache/vibecorp/plans/**`）

## 1. 動作主語で語る

主語と語尾を以下で固定する。

- **主語**: 「このソフトウェア／このフック／このスキル／このエージェント」にする。
- **時制**: 「〜になった／〜できるようになった／〜しなくなった／〜するようになった」を使う。
- **コード参照**: 動作説明の **後ろ** に `📍 根拠:` で最小限に添える。

### 🔄 Before / After

| 文面 | 🔴 Before（コード主語・NG） | 🟢 After（動作主語・OK） |
|---|---|---|
| Issue | `knowledge_buffer.sh` の `flush_buffer()` を修正 | ✅ 知見バッファが PR 作成後に自動で空になるようになった |
| PR | `protect-files.sh` の正規表現を変更 | 🔒 `settings.json` を誤って編集しようとすると自動で止まるようになった |
| 応答 | `get_config()` を置換 | ⚠️ 設定取得が未設定時に落ちていた → デフォルト値を返すようになった |

## 2. マークダウンと状態絵文字

見出し・箇条書き・テーブルで一覧性を出す。

以下の絵文字で状態を示す。

| 絵文字 | 意味 |
|--------|------|
| ✅ | 完了・成功 |
| ⚠️ | 注意・警告 |
| ❌ | 失敗・却下 |
| 🚀 | リリース・デプロイ |
| 🔒 | セキュリティ |
| 🐛 | バグ |
| ✨ | 新機能 |
| 📖 | ドキュメント |
| 🔄 | リファクタ |
| 🧪 | テスト |
| 📍 | 根拠・位置参照 |
| 🤖 | エージェント |

### ⏱️ 30 秒ルール

CEO が 30 秒読んで「何が変わるか」を掴めたら合格。

掴めなければ書き直す。

## 3. 判定フロー

文面を書く前・書いた後に以下 3 段を必ず通す。

1. **読み手は CEO か？** → Yes なら本規約を適用する（No なら対象外として正確性優先）
2. **主語が動作になっているか？** → ソフトウェアやフック等の挙動を主語にする
3. **30 秒で掴めるか？** → マークダウン整形と状態絵文字で変化が一目で分かる

3 つ全てを満たしたら OK。

満たさなければ書き直す。
