# 📖 ドキュメント作成基準

> [!IMPORTANT]
> **可読性が最優先**。マークダウンと絵文字を最大限活用する。
> 読者がスクロールするだけで要点を掴めるように書く。
> CEO・利用者向け文面は `communication.md` の動作主語ルールを守る。

リポジトリ内のドキュメントを書くための基準を定める。

対象種別:

- プロジェクト紹介
- 仕様書
- 設計判断
- セキュリティ説明
- 変更履歴
- プロジェクト規範 等

配置・言語ルールは `documentation.md` を参照する。

本ルールは「書き方」（どう書くか）を扱う。

## 🎯 対象範囲

本ルールは frontmatter の `paths` で対象を絞る。

- 対象: **全 `.md` ファイル**（具体は frontmatter 参照）。
- 読込タイミング: 対象ファイル編集時に lazy-load。
- 公式仕様: https://code.claude.com/docs/en/memory#path-specific-rules

種別ごとの差し分けは末尾「ドキュメント種別ごとの差し分け」を参照。

LLM 向けプロンプト系 `.md` にも本ルールは適用される。

- 対象例: スキル / エージェント / ルール / プロジェクト共通指示 / プロジェクト規範 等
- 追加要件は `prompt-writing.md` が **拡張** として上乗せ。
- 構造: 基底（本ルール）+ 拡張（`prompt-writing.md`）の 2 層。

### 🚫 対象外（paths がマッチしないため本ルールは読み込まれない）

- コード内コメント（`.md` 拡張子ではない）
- 一時ファイル・キャッシュ（リポジトリ外）
- LLM プロンプトファイルのうち `.md` 以外（YAML / TOML 等）

## 👥 読者像と動作主語

書く前に **読者像を 1 行で言語化** する。

- 読者像が変われば語彙・抽象度・主語の取り方が変わる。
- 主語の取り方は下表を参照。

| 読者像 | 主語の取り方 | 語彙の例 |
|---------|--------------|---------|
| **👔 CEO**（経営者） | このソフトウェアが何を実現するか（動作主語） | 「〜できるようになる」「〜が止まる」 |
| **🛠️ 利用者**（プロジェクトを導入する開発者） | このスキル / フックが何を提供するか（動作主語） | 「`/ship` は〜を実行する」 |
| **💻 開発者**（コントリビューター） | この構造 / 設計が何を保証するか（構造主語） | 「責務分離」「依存方向」「拡張点」 |
| **🌍 外部 OSS 読者** | このプロジェクトが何を解くか（プロジェクト主語） | 「Issue 駆動」「Claude Code エコシステム」 |

CEO / 利用者向けは動作主語ルールを必ず守る。

- 根拠: `communication.md`
- 開発者向けは構造を主語にしてよい。
- ただし形容詞止まり（「より良い」「見通しの良い」）は避ける。

## ⏱️ 30 秒スキャン原則

読者が **30 秒スクロールして要点を掴めたら合格**。

掴めなければ書き直す。

スキャン性を高める手段:

| 要素 | 役割 |
|------|------|
| `> [!IMPORTANT]` / `> [!NOTE]` / `> [!WARNING]` 冒頭コールアウト | 最重要メッセージを冒頭で示す |
| 見出し絵文字（1 つ） | 意味のグルーピング標識 |
| テーブル | 比較・前後対照・列挙の整理 |
| 状態絵文字（✅ / ⚠️ / ❌ / 🚀 / 🔒 等) | 状態の即時識別 |
| 太字の体言止め見出し → 改行 → 簡潔な説明 | 段落の入口を明示 |
| Before / After 対比 | 変化の主役を可視化 |

状態絵文字の意味は `communication.md` の絵文字表に従う。

## ✅ 指針（MUST）

ドキュメントを書く前・書いた後に以下 7 項目を必ず通す。

1. 🎯 **読者像を冒頭で固定する**
   - 「このドキュメントは誰のためか」を冒頭で明示する。
   - 読者像によって語彙・抽象度・主語を切り替える。
   - 1 つのドキュメントで複数の読者像を混在させない。
     - 混在するならセクションを分ける。

2. 👁️ **スキャンで要点が掴める構造にする**
   - 冒頭コールアウトで最重要メッセージを示す。
   - 見出し絵文字・テーブル・状態絵文字でスキャン性を確保。
   - 30 秒スクロールで要点を掴めるかを自問する。

3. 📏 **1 文を 50 文字以下に収める**
   - 句点で区切れない長文は読み解きが遅い。
   - 副節や条件節があるなら 1 文を分割する。
   - 主述が遠く離れる構造は避ける。

4. ↩️ **文末（句点）で必ず改行する、または箇条書きをネストする**
   - 連続した文を同じ段落に詰め込まない。
   - 1 文 = 1 行、または 1 文 = 1 箇条書きを基本にする。
   - 詳細が複数階層あるならネスト箇条書きで表現する。

5. 🎨 **マークダウン・絵文字を最大限活用する**
   - 見出し `##` / `###` でセクションを切る。
   - 比較・対照・列挙はテーブルで整理する。
   - 重要な情報はコールアウトで示す。
     - `> [!IMPORTANT]`: 最重要メッセージ。
     - `> [!NOTE]`: 補足情報。
     - `> [!WARNING]`: 警告。
   - 状態絵文字は意味を持たせて使う（飾りで使わない）。

6. 🧭 **設計判断は理由とセットで残す**
   - 「なぜこの構造・この語彙・この見出しか」を書く。
   - 別案を却下した場合は、その理由も短く添える。
   - 仕様変更を伴う更新は Before / After で示す。

7. 🗣️ **汎用語彙で書く**
   - 関数名・クラス名・ファイルパスを地の文で羅列しない。
     - 必要なら `📍 根拠:` で末尾に添える。
   - プロジェクト固有の略語・俗語は初出箇所で必ず定義する。
   - 形容詞止まり（「より良い」「見通しの良い」）は避ける。
     - 代わりに「何が・どこから・どこへ・どう動いたか」を構造で示す。

## ❌ 禁止パターン

以下 7 項目はドキュメントとして機能しない書き方。

書いた後に必ず通す。

- ❌ **実装詳細の羅列で終わる**
  - 関数名・クラス名・ファイルパスを地の文に並べただけになる。
  - 読者像にとっての意味が示されていない。
  - 「`foo.sh` の `bar()` を修正した」だけで終わる。
    - 「何ができるようになったか」が無い。

- ❌ **形容詞止まり**
  - 「より良い」「見通しの良い」「クリーンな」だけで終わる。
  - 構造主語でも「どこから・どこへ・どう動いたか」を構造で示せていない。

- ❌ **1 段落に複数論点**
  - 1 段落で 2 つ以上の主張を扱う。
  - 論点が増えるならサブリストか別段落に分ける（1 項目 1 論点）。

- ❌ **装飾だけの絵文字**
  - 意味を持たない場所に絵文字を散りばめる。
  - 状態絵文字を使うなら `communication.md` の意味に従う。
    - ✅ 完了。
    - ⚠️ 注意。
    - ❌ 失敗 等。

- ❌ **暗黙の前提を残す**
  - 読者像が知らない概念・略語・俗語を定義せずに使う。
  - 関連ファイル・関連 Issue・前段ドキュメントへの参照を貼らない。

- ❌ **1 文 50 文字超の長文**
  - 副節や条件節を連結して 1 文を長文化する。
  - 主述の距離が遠くなり読み解きが遅くなる。

- ❌ **複数文を改行せず連結**
  - 1 段落に句点で区切られた複数文を平文で詰め込む。
  - 箇条書きやテーブルで構造化していない。

## 📚 ドキュメント種別ごとの差し分け

種別ごとに「主語」「冒頭で示すべきもの」「重視軸」が変わる。

| 種別 | 主語の取り方 | 冒頭で示すべきもの | 重視軸 |
|------|------------|------------------|--------|
| **📘 プロジェクト紹介**（README 等） | プロジェクト主語（このプロジェクトが何を解くか） | プロジェクトの一言定義・スクリーンショット / バッジ・最短 Quick Start | OSS 入り口としての敷居の低さ |
| **📋 仕様書** | 動作主語または構造主語（読者像に応じる） | 対象範囲（何を扱う / 扱わない）・前提知識・関連仕様へのリンク | 正確性・網羅性 |
| **⚖️ 設計判断**（ADR 等） | 構造主語（この設計が何を保証するか） | 決定の一言要約・採用した選択肢・却下した別案 | 別案との比較と却下理由 |
| **🔒 セキュリティ説明** | 動作主語（このソフトウェアが何を防ぐか） | 守る対象・想定する攻撃経路・残存リスク | 攻撃経路の網羅と残存リスクの開示 |
| **🔄 変更履歴**（CHANGELOG 等） | 動作主語（リリースで何が変わったか） | バージョン番号・リリース日・カテゴリ別変更点 | リリース間の差分の追跡性 |
| **🌟 プロジェクト規範**（MVV 等) | プロジェクト主語（このプロジェクトの存在意義） | Mission / Vision / Values の 3 層構造 | 判断の最終拠り所として揺るがないこと |

種別と主語を機械的に対応させない。

読者像を 1 行で言語化してから選ぶ。
