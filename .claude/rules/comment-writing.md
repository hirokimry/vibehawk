# 💬 GitHub コメント作成基準

> [!IMPORTANT]
> **GitHub コメントは CEO への報告窓口**。動作主語と 30 秒ルールを守る。
> マークダウンと状態絵文字でスキャン性を上げる。
> 機械生成 Bot 通知は別節の例外規定に従う。コード内コメントは `code-comments.md` が担う。

GitHub 上で投稿するコメント 3 種の書き方を定める。

対象種別:

- 💬 Issue / PR コメント（人間 + エージェント発）
- 🔍 レビューコメント（PR レビューの個別指摘）
- 🤖 Bot 通知コメント（github-actions 等の機械投稿）

書き方の本質は **「読んで 30 秒で何が変わるかが分かる」** こと。

## 🎯 対象範囲

本ルールは frontmatter の `paths` で対象を絞る。

- 対象: GitHub コメントを **生成・投稿する文脈** で編集する各種ファイル。
- 読込タイミング: 対象ファイル編集時に lazy-load。
- 公式仕様: https://code.claude.com/docs/en/memory#path-specific-rules

| 経路 | 対象ファイル例 |
|------|--------------|
| Issue / PR テンプレート | `.github/ISSUE_TEMPLATE/**/*.md` / `.github/pull_request_template.md` |
| Bot 投稿ワークフロー | `.github/workflows/**/*.yml` / `.github/scripts/**/*.sh` |
| コメント投稿スキル | `skills/**/SKILL.md`（`gh issue/pr comment` を含むもの） |

### 🚫 対象外

以下は本ルールの対象外。

- コード内コメント（`.sh` / `.js` / `.ts` / `.py` / `.rb` 等）。
  - 対象基準は `code-comments.md`（旧: `comments.md`、Issue #652）。
- CEO 対話応答 / Issue 本文 / PR 本文 / コミットメッセージ。
  - 対象基準は `communication.md`。
- ドキュメント本体（README / docs/ 等）。
  - 対象基準は `document-writing.md`。
- LLM 向けプロンプト本体（スキル / エージェント / ルール 等）。
  - 対象基準は `prompt-writing.md`。

## 👥 読者像

GitHub コメントは **CEO + エージェント** の両方が読む。

- CEO は「何が変わったか / 何が止まったか」を 30 秒で掴みたい。
- エージェントは判定・再開のための材料として読む。
- どちらにとっても「動作主語」と「状態絵文字」が共通の最大公約数となる。

| 読者 | 何を期待しているか |
|------|------------------|
| 👔 CEO | ソフトウェアの挙動変化を即時把握する |
| 🤖 エージェント | 次に取るべき行動の判断材料を得る |

両者が読む前提で、**動作主語 + 状態絵文字 + マークダウン整形** を必ず満たす。

## 🎨 状態絵文字

冒頭または見出しに状態絵文字を 1 つ付ける。

意味は `communication.md` の絵文字表に従う。

| 絵文字 | 意味 |
|--------|------|
| ✅ | 完了・成功 |
| ⚠️ | 注意・警告 |
| ❌ | 失敗・却下 |
| 🚀 | リリース・デプロイ |
| 🔒 | セキュリティ |
| 🐛 | バグ |
| ✨ | 新機能 |
| 📖 | ドキュメント |
| 🔄 | リファクタ |
| 🧪 | テスト |
| 📍 | 根拠・位置参照 |
| 🤖 | エージェント |

装飾だけの絵文字は使わない。

## ⏱️ 30 秒ルール

CEO がコメント全体を 30 秒で読み切れることを目標にする。

- 冒頭 1 行で結論を示す。
- マークダウン整形で要点をスキャン可能にする。
- 長文化するならテーブル / 箇条書きで分解する。

掴めなければ書き直す。

## 🧭 判定フロー

コメントを書く前・書いた後に以下 3 段で必ず通す。

1. **読み手は CEO か？** → Yes なら本規約を適用する。
2. **主語が動作になっているか？** → ソフトウェアの挙動変化を主語にする。
3. **30 秒で掴めるか？** → 状態絵文字と整形で変化が一目で分かる。

3 つを満たさなければ書き直す。

## ✅ 指針（MUST）

コメントを書く前・書いた後に以下 6 項目を必ず通す。

1. 🗣️ **動作主語で語る**
   - 主語は「このソフトウェア / このフック / このスキル」。
   - 時制は「〜になった / 〜できるようになった / 〜しなくなった」。
   - コード参照は本文の **後ろ** に `📍 根拠:` で添える。

2. 🎯 **冒頭 1 行で結論を示す**
   - 状態絵文字 + 1 文で「何が起きたか」を示す。
   - 詳細は次行以降に展開する。
   - 読者は冒頭 1 行で次の行動を判断できる。

3. 🎨 **マークダウンと状態絵文字を活用する**
   - 見出し `##` / `###` でセクションを切る。
   - 比較・対照・列挙はテーブルで整理する。
   - 状態絵文字は意味を持たせて使う。

4. 📏 **1 文 50 文字以下に収める**
   - 副節や条件節があるなら 1 文を分割する。
   - 句点で必ず改行する。
   - 主述が遠く離れる構造は避ける。

5. 🧭 **設計判断は理由とセットで残す**
   - 「なぜこの修正 / この却下か」を書く。
   - 別案を却下した場合は理由を短く添える。
   - 仕様変更を伴う指摘は Before / After で示す。

6. 🔗 **関連 Issue / PR / コミットへ参照を貼る**
   - `#N` 形式または短縮 URL で示す。
   - 別 Issue へ送る場合は新規 Issue 番号を明示する。
   - 同じ議論を別所に再掲しない。

## ❌ 禁止パターン

以下 6 項目はコメントとして機能しない書き方。

書いた後に必ず通す。

- ❌ **コード主語で語る**
  - 「`foo.sh` の `bar()` を修正」だけで終わる。
  - 何ができるようになったかが示されていない。
  - 動作主語に書き直す。

- ❌ **進捗報告だけのコメント**
  - 「対応中です」「あとで見ます」だけで終わる。
  - 設計・判断・成果のいずれも残っていない。
  - 状態が分かる必要がない情報は投稿しない。

- ❌ **形式だけの定型挨拶**
  - 「お疲れ様です」「ご確認ください」だけで埋める。
  - 読者は判定材料を得られない。
  - 結論を冒頭 1 行で示す。

- ❌ **1 段落に複数論点**
  - 1 段落で 2 つ以上の指摘を扱う。
  - 論点が増えるなら別段落 / 別コメントに分ける。
  - レビューコメントは 1 コメント 1 論点が原則。

- ❌ **装飾だけの絵文字**
  - 意味を持たない位置に絵文字を散りばめる。
  - 状態絵文字は意味を持たせて使う。
  - 飾り目的の絵文字は外す。

- ❌ **暗黙の前提を残す**
  - 関連 Issue / PR / コミットへの参照を貼らない。
  - プロジェクト固有の略語を初出で定義しない。
  - 読者が文脈を別所から推測する必要を残さない。

## 💬 Issue / PR コメント

Issue / PR 本体への追加コメントを書く時の差し分け。

- 主語: 動作主語（このソフトウェア / フック / スキル）。
- 冒頭で示すべきもの: 状態絵文字 + 1 行結論。
- 重視軸: 「何が変わるか / 何が止まるか」を最短で渡す。

### ✅ 書き方の例

```markdown
✅ 計画レビューが 2 回ループで通過した

- 1 回目: テスト項目欠落 → 補強した
- 2 回目: 残課題 0 件で確定した

📍 根拠: `~/.cache/vibecorp/plans/<repo-id>/dev_631_...md`
```

### ❌ 書いてはいけない例

```markdown
plan-review-loop.sh の reviewIssues 配列を空にする
処理が走ったので OK だと思います。
```

何ができるようになったかが示されていない。

## 🔍 レビューコメント

PR レビューの個別指摘を書く時の差し分け。

- 主語: 動作主語または構造主語（指摘対象に応じる）。
- 冒頭で示すべきもの: severity マーカー + 結論 1 行。
- 重視軸: 修正対象か管轄外かを `review-handling.md` の基準で示す。

### severity マーカー

severity の表記は `severity/coderabbit.md` を Source of Truth として採用する。

| 絵文字 | severity |
|--------|---------|
| 🔴 | Critical |
| 🟠 | Major |
| 🟡 | Minor |
| 🔵 | Trivial |
| ⚪ | Info |

### ✅ 書き方の例

```markdown
🟠 Major: `gh api` のページネーション漏れ

`comments` エンドポイントは 30 件超で欠落する。

- 修正: `--paginate` フラグを付ける
- 根拠: `.claude/rules/shell.md` ページネーション節

intent/bugfix の重視軸（修正範囲の妥当性）に該当する。
```

### ❌ 書いてはいけない例

```markdown
LGTM。ところで `foo.sh` の命名って改善できそう。
```

severity も結論もなく、議論が拡散する。

## 🤖 Bot 通知コメント

`github-actions` 等の自動投稿で使うコメントの差し分け。

- 主語: 動作主語（ワークフロー / Bot が何を検知したか）。
- 冒頭で示すべきもの: 状態絵文字 + 検知結果 1 行。
- 重視軸: 機械的に同じテンプレを再現できること。

### ✅ 書き方の例（人手でテンプレを設計する場合）

```markdown
❌ intent ラベルが付いていません

この Issue には `intent/*` ラベルを 1 つだけ付与してください。

📍 根拠: `.claude/rules/intent-labels.md`
```

### 🤖 機械生成テンプレートの例外

以下のケースは「動作主語」適用の対象外とする。

- GitHub ネイティブの自動投稿（`Closes #N via PR #M` 等）。
- 第三者 Bot の固定テンプレ（CodeRabbit / Dependabot 等）。
- ワークフローが構造化通知として投稿する短文。
  - 例: `coverage: 87.3% (+0.4%)`
  - 例: `built artifacts: foo.tgz, bar.tgz`

機械生成テンプレートは **動作主語ではなく構造化通知** で良い。

ただし以下は守る。

- 状態絵文字を 1 つは付ける（✅ / ⚠️ / ❌ / 🚀 等）。
- 必要なら関連 Issue / PR を `#N` で示す。
- 詳細リンク（CI ログ等）は本文末尾に貼る。

### ❌ 書いてはいけない例

```markdown
ワークフローが完了しました。
```

何が完了したか・成功か失敗かが示されていない。
