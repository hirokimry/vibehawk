# 🧪 テスト追加規約

> [!IMPORTANT]
> hooks やスクリプトを追加・変更した場合、`tests/` 配下に **対応するテストを必ず書く**。
> テストファイルは `test_*.sh` 命名規則に従い、CI で自動実行される前提で書く。
> `set -euo pipefail` 下で前提ファイル不在時は `fail()` の後に **明示的に `exit 1`** する。

hooks / スクリプト変更に対するテスト追加義務と、`set -euo pipefail` 下で発生しやすい早期終了・リソース解放の落とし穴を集約した規約。

## ✅ 追加義務

hooks やスクリプトを追加・変更した場合、`tests/` 配下に対応するテストを書く。

- テストファイルは `test_*.sh` の命名規則に従う
- テストは CI で自動実行される前提で書く
- 既存テストが壊れていないか確認してからコミットする
- 新しい hook を追加したらテストケースも同時に追加する

## 🛑 `set -e` 下でのテストスクリプトの早期終了

`set -euo pipefail` を使ったテストスクリプトで前提ファイルが存在しない場合は、`fail()` の呼び出しだけでは不十分。`fail()` は `FAILED` カウンタを増やすだけで `exit` しないため、後続のテストが不正な状態で継続する。

前提ファイル不在のように後続テストが全て無意味になる場合は、`fail()` の後に明示的に `exit 1` する。

```bash
if [[ -f "$SKILL_FILE" ]]; then
  pass "SKILL.md が存在する"
else
  fail "SKILL.md が存在しない"
  # 前提ファイル不在 → 後続テストは全て無意味なので即終了
  exit 1
fi
```

## 🧹 trap cleanup EXIT でのリソース解放

`set -euo pipefail` 下のテストスクリプトで `trap cleanup EXIT` を使う場合、cleanup 内のリソース解放コマンドはテスト結果に影響させてはならない。

- 一時ファイル・一時ディレクトリ・バックグラウンドプロセス・外部リソース（コンテナ、マウント、ネットワーク等）の解放は、失敗しても終了コードに影響させない
- 各解放コマンドの末尾に `|| true` を付けて失敗を無害化する
  - リダイレクト `>/dev/null 2>&1` 単独では終了ステータスを抑制できない
  - サブシェル `( set +e; ... )` も `set -e` 下の `trap EXIT` 内では非ゼロ終了コードを伝播させる（外側の `set -e` が trap 実行時の失敗を検出するため）
  - `|| true` のみが確実に終了コードを 0 に保つ

```bash
cleanup() {
  rm -rf "$TMPDIR_ROOT" || true
}
trap cleanup EXIT
```

この失敗抑制はテストのフォールバック禁止ルールの対象外。cleanup の目的はリソース解放であり、失敗しても後続テストへの影響はないため。
