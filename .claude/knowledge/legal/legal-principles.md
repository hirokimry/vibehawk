# 法務判断原則

docs/POLICY.md のポリシーと MVV.md から導出される、法務分析員の判断基準。
ポリシーの詳細は `docs/POLICY.md` を参照すること。

## MVV との対応（2026-Q2 制定）

vibehawk の法務判断は MVV の以下の Value を設計レベルの根拠として持つ。

| Value | 法務上の含意 |
|-------|-------------|
| Value 1「利用者の契約だけで、完結させる」 | LLM プロバイダー等の「再販・代理利用禁止」条項への抵触を排除する設計根拠 |
| Value 2「観察する、書き換えない」 | GitHub API の書き込み権限濫用リスクを設計レベルで排除する根拠 |
| Value 4「公式の道を、迂回せず歩く」 | スクレイピング・非公式 API 使用禁止の根拠。外部サービス ToS 遵守を宣言 |

## POLICY.md 5 大方針の法務上の含意（2026-Q2 ship #6 確認）

POLICY.md に定義された 5 大方針は、各方針が法務上の根拠を持つ。

| 大方針 | 法務上の含意 |
|--------|-------------|
| 大方針 1「カスタムは外から注入させる」 | 機能非内蔵・カスタム余地の提供に留まる設計。Bot 機能の過剰拡張による ToS 違反リスクを排除 |
| 大方針 2「レビュー & 修正提案まで」 | コード自動生成・自動 commit の明示禁止。利用者合意なくコードを増やす bot 行動の ToS 制限と整合 |
| 大方針 3「severity 判定はツール側、捌き方は利用者側」 | コスト制御の例外的ツール側保持は初心者保護目的の意図的例外。「捌き方は利用者側」原則の侵害には非該当 |
| 大方針 4「状態管理は GitHub に置く、専用 DB は持たない」 | 個人情報を蓄積するインフラ不持 = 個人情報保護法・GDPR データ最小化原則の設計実装。漏洩リスクを構造的に排除 |
| 大方針 5「PR メタデータ操作はしない」 | Bot による label 付与・milestone 設定等の禁止。GitHub ToS 上の Bot 行動ガイドライン（意図しない副作用回避）との整合 |

## 最重要制約

<!-- プロジェクトに合わせて法的制約を記述する -->

1. **ライセンス遵守**: 依存パッケージのライセンスがプロジェクトのライセンスと互換であること
2. **個人情報保護**: 個人情報の収集・保存・処理は POLICY.md の方針に従うこと
3. **利用規約遵守**: 外部サービスの利用規約に違反しないこと（Value 4 を根拠とする）

## 判断の優先順位

法的リスクが競合する場合の優先順:

<!-- プロジェクトに合わせて優先順位を記述する -->

1. ユーザーデータの保護（個人情報・機密情報）
2. ライセンス整合性（OSS ライセンス違反の防止）
3. 利用規約の遵守（外部サービスの TOS）
4. 知的財産権の保護

## レビュー時の判断基準

### 即ブロック

<!-- 問答無用で差し戻すべき法的リスクを記述する -->

- ライセンス非互換の依存パッケージの追加
- 個人情報を暗号化せずに保存・送信する変更
- 外部サービスの利用規約に明示的に違反する変更

### 要確認（CLO へエスカレーション）

<!-- CLO の判断が必要なケースを記述する -->

- 新しい外部サービス・API の導入（利用規約の確認が必要）
- ライセンスがグレーゾーンの依存パッケージ
- 個人情報の取り扱い範囲が変わる変更

### 許容

<!-- 追加確認なしで承認できるケースを記述する -->

- MIT / BSD / Apache 2.0 ライセンスの依存パッケージ追加
- 個人情報を扱わないコード変更
- 既存の法的枠組み内での変更

## エスカレーション基準

以下の場合は必ず CLO にエスカレーションする:

- 判断に不確実性がある場合（ライセンスの解釈が曖昧等）
- 合議で意見が割れた場合
- 新しい法的領域に踏み込む変更の場合
- POLICY.md に記載のないケースに遭遇した場合
