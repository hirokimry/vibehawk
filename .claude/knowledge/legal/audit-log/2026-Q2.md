### 対象
vibehawk MVV（Mission / Vision / Values）制定に伴う CLO 観点レビューと知見反映。

### 法務観点評価

**Mission「レビューツールに追加課金が要らない世界をつくる」**
- 商標リスク: 既存製品名（CodeRabbit / GitHub / Greptile 等）を含まず、特定製品への誹謗・誤認を誘発する文言なし。
- 反競争的表現: 機能訴求であり、競合製品への虚偽広告・誹謗には該当しない。
- 判定: 問題なし。

**Value 1「利用者の契約だけで、完結させる」**
- LLM プロバイダー等の「再販・代理利用禁止」条項の対象になりうる設計を排除する思想。
- vibehawk が利用者の API キーを横取りしない設計の法務上の根拠となる。

**Value 2「観察する、書き換えない」**
- GitHub API の書き込み権限濫用リスクを設計レベルで排除。
- PR メタデータの読み取りのみ行い書き換えを行わない制約は ToS 遵守上の重要な設計制約。

**Value 4「公式の道を、迂回せず歩く」**
- スクレイピング・非公式エンドポイント禁止を Value として宣言。
- GitHub / Anthropic / その他外部サービスの ToS 遵守を設計思想レベルで担保。
- legal-principles.md の「外部サービスの利用規約に明示的に違反する変更 → 即ブロック」の判断基準の根拠となる。

### 参照した法規制・ガイドライン
- GitHub Terms of Service（API 利用規約）
- Anthropic Usage Policy（再販・代理利用禁止条項）
- 商標法（スローガン的フレーズの類似性評価）

### 変更したファイル

- `.claude/knowledge/legal/legal-principles.md`: MVV との対応テーブルを追加

### POLICY.md 5 大方針との整合評価

**大方針 1「カスタムは外から注入させる」**
- `.vibehawk.yaml` オーバーライド機構（cost-analysis.md に記載）は「機能を内蔵せず、カスタム指示の余地だけ残す」方針と完全整合。
- 利用者が外から注入する設計思想であり、機能内蔵の拡大解釈にはあたらない。
- 法務上の問題なし。

**大方針 2「レビュー & 修正提案まで、コード生成は絶対にしない」**
- specification.md に「やらない範囲」として明示。コード自動生成・自動 commit を禁止する旨が文書化されている。
- 利用者の合意なくコードを増やす行為の排除が明文化されており、ToS 上の bot 行動制限との整合を担保する。
- 法務上の問題なし。

**大方針 3「severity 判定はツール側、捌き方は利用者側」**
- コスト制御を初心者保護の安全弁として例外的にツール側に持つ設計（cost-analysis.md）は、大方針 3 の例外として意図的に選択されたもの。
- 利用者保護目的の例外であり、「捌き方は利用者側」原則の侵害には該当しない。
- 法務上の問題なし。

**大方針 4「状態管理は GitHub に置く、専用 DB は持たない」**
- specification.md と ai-organization.md の 2 ファイルで「専用 DB を持たない」設計を明示。
- 個人情報を蓄積するインフラを持たない設計は、個人情報保護法・GDPR 等の観点で「収集しない = 漏洩リスクなし」を成立させる。
- 専用サーバー・ベクタ DB 不持の宣言は legal-principles.md の「個人情報保護」制約の設計レベルでの実装根拠となる。
- 法務上の問題なし。

**大方針 5「PR メタデータ操作はしない」**
- 「やらない範囲」として specification.md に明示。label 付与 / milestone 設定 / description 補完 / assignee 操作の禁止が文書化。
- GitHub API の書き込み権限濫用リスクを設計上排除（Value 2「観察する、書き換えない」とも整合）。
- GitHub Terms of Service 上、Bot による PR メタデータ操作は利用者リポジトリへの意図しない副作用リスクがある。明示的に禁止する方針は ToS 遵守上の重要な設計制約。
- 法務上の問題なし。
