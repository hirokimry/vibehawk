## 2026-06-13 — Issue #346 / PR #348 配布 workflow 自己完結化（外部リポジトリで初めてプロダクトが成立）

### 何を判断したか

`npx vibehawk setup` で外部リポジトリに配布した workflow が最初の step で全滅する欠陥（v0.2.2 まで公開状態で潜伏）の修正（PR #348）を踏まえ、配布プロダクトとしての構造的盲点と再発防止のレビュー観点を記録した。

### どう判断したか

**確定した認識 3 点:**

1. **「自リポジトリで動く」は「外部で動く」を保証しない。** dogfooding（`templates/` と `.github/workflows/` のテンプレート完全一致運用）は、参照スクリプト約 37 ファイルが vibehawk リポジトリに存在するため、外部リポジトリでの破綻を構造的に隠す。外部 dogfooding（#56）未実施のまま v0.2.2 まで公開されていた。
2. **導入前検証は発見経路として機能した。** 最初の外部導入候補（vibecorp、hirokimry/vibecorp#531）の事前調査（npm 実物展開）で発覚した。「最初の外部導入＝最初の実地検証」になってしまう構造を、導入前調査が補った。
3. **v0.2.3 は外部成立の節目。** 「外部リポジトリのレビューが本来動作として始まる」= プロダクトとして初めて外部で成立するリリース。以後「外部で動く」が既達成の前提となり、これを壊す変更は回帰となる。

**復旧導線の正確性も判断として記録:** 既配布利用者向けの復旧コマンドは `npx vibehawk@latest install --repo <owner>/<repo> --overwrite` が正。`setup --overwrite` は存在しない。存在しないコマンドを docs に書くと利用者の復旧導線そのものが壊れるため、docs 記載の CLI コマンドは `cli/` 実装と突合して実在を検証する。

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **Mission 整合**: 「レビューツールに追加課金が要らない世界をつくる」は外部リポジトリで動いて初めて実現する。README 保全フレーズ「merge gate の主軸」と実態（外部では check が一度も発火しない）が v0.2.2 まで乖離していた。Mission の打ち出しと実動作の乖離は最重度のプロダクト品質問題。
- **Value 1「利用者の契約だけで、完結させる」**: 配布物が自己完結しない状態は Value 1 の構造的不履行だった。リリースタグ pin 付き 2nd checkout による自己完結化は Value 1 の実現。
- **ユーザー体験第一**: 復旧導線に存在しないコマンドを書くことは、障害時という最も UX が重要な場面での体験破壊。

### 引き継ぎルール（次回レビューへの観点追加）

1. 配布物（`templates/` / `cli/install.js`）に触れる変更は「外部リポジトリ相当の状態で動くか」をレビュー観点に必ず含める。自リポジトリ CI green は外部動作の証拠にならない。
2. docs に記載する CLI コマンド・オプションは実装との突合（実在検証）を必須とする。
3. #56（外部 dogfooding）が close されるまで、配布系変更は外部相当シミュレーションテスト（参照スクリプト不在 checkout の再現等）を完了条件に含めることを推奨する。

## 2026-05-08 — MVV 確定（CEO ⇄ CPO 合議・6 ラウンド推敲）

### 何を判断したか

vibehawk の MVV を CEO ⇄ CPO 合議で確定した。試案を起点に 6 ラウンドの推敲を経て以下に確定した:

- **Mission**: レビューツールに追加課金が要らない世界をつくる。
- **Vision**: 一度の支払いが、つくるすべてに届く世界。
- **Values**:
  1. 利用者の契約だけで、完結させる
  2. 観察する、書き換えない
  3. 指摘する、強制しない
  4. 公式の道を、迂回せず歩く

### どう判断したか

試案「追加サブスク負担ゼロで成立させる」型から「状態描写型」へ全面推敲した。Values は試案の 3 動詞並列から「対象+動作」の 2 部構造に再構築した。

主要な判断ポイント:

1. **Mission/Vision は状態描写型に振る**: 「〇〇する」型は手段に縛られて陳腐化する。「〇〇な世界をつくる」型は状態を語り 5〜10 年スパンで意味を保つ
2. **Values は動詞起点・2 部構造**: 「観る、伝える、持たない」型の 3 動詞並列は目的語が消えて意味不明になりやすい（CEO から「持たないって何？」と指摘）。「観察する、書き換えない」のように 2 動詞で対象と動作を明示する構造に統一した
3. **「読んだだけで意味がわかる」を必須とした**: 鷹のメタファー等、カッコよさ優先の案は CEO に即却下された。意味の明瞭さが絶対優先
4. **技術原則は Values から分離**: 「状態を持たない」のような設計判断は Values に書かず `docs/design-philosophy.md` に分離した（Stripe / Basecamp / GitLab の業界定石）

### なぜそう判断したか

- **Mission 整合**: vibehawk のコア思想「所有を手放すことで自由になる」が Mission/Vision/Values を貫く軸。LLM を所有しない・状態を所有しない・PR メタデータを所有しない
- **シンプルさ原則**: カッコよさ ≠ 装飾、カッコよさ = 簡潔さ・力強さ・記憶しやすさ。意味の明瞭さを損ねる装飾は排除
- **陳腐化耐性**: 特定の課金形態（サブスク・買い切り等）や技術名に依存しない表現を選択

### Issue 完了条件との整合

Issue #4 完了条件「試案通り」と確定文の乖離が発生した。MVV.md を変更するのではなく、Issue 本文を確定文側に更新することで完了条件と実態を一致させた。CEO が起票者のため Issue 本文編集は権限内。

## 2026-05-08 — ship #6 README 乖離の継続放置（3 PR 連続パターン）

## 2026-05-08 — ship #6 specification.md アーキテクチャ章の構成方針

### 何を判断したか

specification.md のアーキテクチャ章に bash サンプルコードが含まれており、CTO から「保守性懸念」（Trivial）の指摘があった。

アーキテクチャ章は以下 6 サブセクションで構成される:
- 状態管理 / メタデータ仕様 / マルチリポジトリ対応 / インクリメンタルレビュー / sticky review state / @mention チャット応答
- 配置: 「やらない範囲」セクション直後・「非機能要件」前

### どう判断したか

bash サンプルはあくまで「実装パターンのイメージ補助」として掲載されている。保守性懸念（Trivial）は受け入れるが、現時点では削除ではなく「spec 内サンプルは参照的なものであり実装と完全同期しないことがある」を明示することで対応を検討する。

**今後の方針**:
- サンプルコードを spec に書く場合は「実装の参考（非規範的）」と注記する
- 規範的な実装フローは spec ではなく別ドキュメント（設計ドキュメント等）に分離することを推奨する

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **シンプルさ原則**: spec に実装コードが混在すると「spec の変更が実装を縛る」か「実装の変更が spec を古くする」のどちらかが発生する
- **一貫性原則**: spec は「何をするか（What）」を記述するドキュメント。「どうやるか（How）」が混在すると責務が曖昧になる

## 2026-05-08 — Vision 案 A-1 の表現推敲（「払った AI」→「一度の支払いで AI が」）

### 何を判断したか

Vision 候補 A-1「一度払った AI が、開発のすべての工程に行き渡る世界。」の前半表現について、CEO フィードバック（「AI を払う／買うは日本語として不自然」）を受け、最終推奨案として **B-1「一度の支払いで AI が、開発のすべての工程に行き渡る世界。」** を提示した。

### どう判断したか

不自然さの本質: 「払う」の目的語は対価・金銭であり、サービス主体である AI を直接目的語に取ると格関係が崩れ、物化・擬人化のニュアンスが生じる。

推敲 3 案を提示:

- B-1: 一度の支払いで AI が、開発のすべての工程に行き渡る世界。（最終推奨）
- B-2: AI への対価が一度で済み、開発のすべての工程に行き渡る世界。
- B-3: 一度払えば AI が、開発のすべての工程に行き渡る世界。

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **Mission 整合**: 「レビューツールに追加課金が要らない世界をつくる」の核である「二重支払い否定」を「一度の支払いで」が直接表現する。
- **シンプルさ原則**: 3 案中で最短かつ語順が自然。AI が主語に立つことで「AI が行き渡る」という後半（CEO 承認済み）と主語が連続する。
- **陳腐化耐性**: 特定の課金形態（サブスク・買い切り等）や技術名に依存せず、5-10 年スパンで意味が保たれる。
- **擬人化・物化回避**: 「払った AI」「買った AI」のような違和感を排除。
- **CEO 承認済み後半の保持**: 「開発のすべての工程に行き渡る世界」は変更せず、前半のみ推敲。

## 2026-05-30 — Issue #230 vibehawk インライン指摘フォーマットを CodeRabbit 完全模倣（方針 A）に確定

### 何を判断したか

vibehawk のインライン指摘コメント（`comments[]`）のフォーマットを **方針 A（CodeRabbit 完全模倣）** に確定した（Issue #230）。並走時に CEO が「同じ商品」と感じる完成度を狙う。

確定した構成要素（実測に基づく固定形式）:

- **3 軸ラベル**: `_<カテゴリ>_ | _<severity>_ | _<労力>_`（イタリック・パイプ区切り）。
- **太字タイトル + 説明段落**の 2 部構成。
- **🤖 AI 向け修正指示**の `<details>` 折り畳み（中身は日本語・vibehawk 文面、英語定型文の literal コピーはしない）。
- **📝 Committable suggestion** 折り畳み（suggestion がある指摘のみ）。
- **`<!-- vibehawk:inline -->` 識別フッタ**（CodeRabbit 文言の literal コピーは出所を偽るため禁止）。
- **コールアウト（`> [!CAUTION]` 等）は模倣しない**（実測でインラインにはほぼ 0%）。

### どう判断したか

過去 PR の CodeRabbit インラインコメントを全件実測した（レビューコメント 357 件中 CodeRabbit 186 件、うちラベル付き指摘 **157 件**）。2 回独立集計で一致した値を採用。

- カテゴリ: ⚠️ Potential issue 154 / 🛠️ Refactor suggestion 3（インラインに Nitpick・Verification は出ない）。
- severity（必ず付く）: 🟠 Major 93 / 🟡 Minor 49 / 🔴 Critical 15。
- 労力（必ず付く・現状把握に無かった新軸）: ⚡ Quick win 147 / 🏗️ Heavy lift 10。
- 本文構造の同梱率: 太字タイトル+説明 100% / `<details>` 100% / 🤖 Prompt for AI Agents 99.4% / auto-generated フッタ 83% / Committable suggestion 22% / コールアウト ほぼ 0%。

方針 A / B（現状維持）/ C（ハイブリッド）から **A** を採用。ただし「枠は再現・中身は日本語/vibehawk 化」とし、CodeRabbit の英語文言・識別子の literal コピーはしない。

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **CEO 判定基準**: 30 秒スキャン性 + CodeRabbit との並走で「同じ商品」と感じられること。完全模倣が最もこれを満たす。
- **Value 2「観察する、書き換えない」**: コールアウト（`> [!CAUTION]`）は強い警告表現だが実測ほぼ 0% のため非模倣とし、過剰表現を避ける。
- **Value 3「指摘する、強制しない」**: 修正提案 Suggestions の自動適用は禁止のまま（Bot 自身は commit しない）。
- **出所の誠実性**: フッタ・🤖 修正指示の文言は vibehawk 製と分かる日本語にし、CodeRabbit 文言の literal コピーで出所を偽らない。
