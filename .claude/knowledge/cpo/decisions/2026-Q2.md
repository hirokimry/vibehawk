# CPO 判断記録 2026-Q2

## 2026-05-08 — MVV 確定（CEO ⇄ CPO 合議・6 ラウンド推敲）

### 何を判断したか

vibehawk の MVV を CEO ⇄ CPO 合議で確定した。試案を起点に 6 ラウンドの推敲を経て以下に確定した:

- **Mission**: レビューツールに追加課金が要らない世界をつくる。
- **Vision**: 一度の支払いが、つくるすべてに届く世界。
- **Values**:
  1. 利用者の契約だけで、完結させる
  2. 観察する、書き換えない
  3. 指摘する、強制しない
  4. 公式の道を、迂回せず歩く

### どう判断したか

試案「追加サブスク負担ゼロで成立させる」型から「状態描写型」へ全面推敲した。Values は試案の 3 動詞並列から「対象+動作」の 2 部構造に再構築した。

主要な判断ポイント:

1. **Mission/Vision は状態描写型に振る**: 「〇〇する」型は手段に縛られて陳腐化する。「〇〇な世界をつくる」型は状態を語り 5〜10 年スパンで意味を保つ
2. **Values は動詞起点・2 部構造**: 「観る、伝える、持たない」型の 3 動詞並列は目的語が消えて意味不明になりやすい（CEO から「持たないって何？」と指摘）。「観察する、書き換えない」のように 2 動詞で対象と動作を明示する構造に統一した
3. **「読んだだけで意味がわかる」を必須とした**: 鷹のメタファー等、カッコよさ優先の案は CEO に即却下された。意味の明瞭さが絶対優先
4. **技術原則は Values から分離**: 「状態を持たない」のような設計判断は Values に書かず `docs/design-philosophy.md` に分離した（Stripe / Basecamp / GitLab の業界定石）

### なぜそう判断したか

- **Mission 整合**: vibehawk のコア思想「所有を手放すことで自由になる」が Mission/Vision/Values を貫く軸。LLM を所有しない・状態を所有しない・PR メタデータを所有しない
- **シンプルさ原則**: カッコよさ ≠ 装飾、カッコよさ = 簡潔さ・力強さ・記憶しやすさ。意味の明瞭さを損ねる装飾は排除
- **陳腐化耐性**: 特定の課金形態（サブスク・買い切り等）や技術名に依存しない表現を選択

### Issue 完了条件との整合

Issue #4 完了条件「試案通り」と確定文の乖離が発生した。MVV.md を変更するのではなく、Issue 本文を確定文側に更新することで完了条件と実態を一致させた。CEO が起票者のため Issue 本文編集は権限内。

## 2026-05-08 — ship #6 C*O 間判断割れ: ai-organization.md 配置問題

### 何を判断したか

ship #6 の実装中に `docs/ai-organization.md` の配置について CPO と CTO の判断が対立した。

- **CPO 推奨（B 案）**: ファイルを移動する（配置の一貫性・管理しやすさの観点）
- **CTO 判定（A 案）**: CEO の当初意図を尊重して現状維持とする

スキル仕様「Issue 完了条件優先」により CTO 判定が確定し、CPO 推奨は採用されなかった。

### どう判断したか

CPO として B 案を推奨したが、スキルのガバナンスルール（Issue 完了条件が判断の拠り所）に従い、CTO 判定（CEO 意図尊重）を受け入れた。

**本事例のパターン**: 「CPO 推奨（構造的整合）vs CTO 判定（CEO 意図尊重）」で後者が優先される。

### なぜそう判断したか（MVV・原則のどれに基づくか）

- CPO 推奨の根拠: プロダクト管轄（docs/ 配置の一貫性）はプロダクト原則「一貫性」に基づく
- CTO 優先の根拠: Issue 完了条件（CEO の意図）が最終拠り所というスキル仕様上のルール

### 次回同種判断へのリファレンス

「CPO が構造的整合を推奨 → CEO 意図（Issue 完了条件）で覆される」pattern は再発する可能性がある。
CPO がこのケースで重要なのは:
1. 推奨を明示的に記録しておく（次回類似ケースで「過去にも同じ判断があった」と参照できる）
2. 採用されなかった推奨も記録の対象（ナレッジは合意した結論だけでなく対立した議論も含む）
3. 「CPO 推奨が覆された」事例が複数蓄積した場合、プロダクト方針の明文化をファウンダーにエスカレーションする

## 2026-05-08 — ship #6 README 乖離の継続放置（3 PR 連続パターン）

### 何を判断したか

ship #4 / #5 / #6 の 3 PR 連続で sync-check が「README 要更新（別 Issue 推奨）」を指摘したが、いずれも CEO 判断で起票なし。

### どう判断したか

CPO として「3 PR 連続放置」を明示的な認知パターンとして記録した（反映不要・記録のみ）。

**判定: CPO 側は追加アクションなし。** 理由:
- README はプロダクトの外部公開ドキュメントであり、内容の正確性はプロダクト方針に影響しうる
- ただし起票判断は CEO のものであり、CPO が単独で強制できる領域ではない
- 3 PR 連続で同じ指摘が出続けることは「技術的負債の認知的コスト」としてパターン登録する価値がある

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **一貫性原則**: 仕様と README のドリフトが長期放置されると、将来の利用者に誤情報を提供するリスクがある（プロダクト品質の観点）
- ただし起票判断は CEO の管轄。CPO の役割は「パターンを認知して記録し、エスカレーション判断の材料を作る」こと

### 閾値の考え方

同種の sync-check 指摘が **5 PR 連続** に達した場合、CPO として「仕様とドキュメントのドリフト」をファウンダーにエスカレーションすることを検討する。

## 2026-05-08 — ship #6 specification.md アーキテクチャ章の構成方針

### 何を判断したか

specification.md のアーキテクチャ章に bash サンプルコードが含まれており、CTO から「保守性懸念」（Trivial）の指摘があった。

アーキテクチャ章は以下 6 サブセクションで構成される:
- 状態管理 / メタデータ仕様 / マルチリポジトリ対応 / インクリメンタルレビュー / sticky review state / @mention チャット応答
- 配置: 「やらない範囲」セクション直後・「非機能要件」前

### どう判断したか

bash サンプルはあくまで「実装パターンのイメージ補助」として掲載されている。保守性懸念（Trivial）は受け入れるが、現時点では削除ではなく「spec 内サンプルは参照的なものであり実装と完全同期しないことがある」を明示することで対応を検討する。

**今後の方針**:
- サンプルコードを spec に書く場合は「実装の参考（非規範的）」と注記する
- 規範的な実装フローは spec ではなく別ドキュメント（設計ドキュメント等）に分離することを推奨する

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **シンプルさ原則**: spec に実装コードが混在すると「spec の変更が実装を縛る」か「実装の変更が spec を古くする」のどちらかが発生する
- **一貫性原則**: spec は「何をするか（What）」を記述するドキュメント。「どうやるか（How）」が混在すると責務が曖昧になる

## 2026-05-08 — Vision 案 A-1 の表現推敲（「払った AI」→「一度の支払いで AI が」）

### 何を判断したか

Vision 候補 A-1「一度払った AI が、開発のすべての工程に行き渡る世界。」の前半表現について、CEO フィードバック（「AI を払う／買うは日本語として不自然」）を受け、最終推奨案として **B-1「一度の支払いで AI が、開発のすべての工程に行き渡る世界。」** を提示した。

### どう判断したか

不自然さの本質: 「払う」の目的語は対価・金銭であり、サービス主体である AI を直接目的語に取ると格関係が崩れ、物化・擬人化のニュアンスが生じる。

推敲 3 案を提示:

- B-1: 一度の支払いで AI が、開発のすべての工程に行き渡る世界。（最終推奨）
- B-2: AI への対価が一度で済み、開発のすべての工程に行き渡る世界。
- B-3: 一度払えば AI が、開発のすべての工程に行き渡る世界。

### なぜそう判断したか（MVV・原則のどれに基づくか）

- **Mission 整合**: 「レビューツールに追加課金が要らない世界をつくる」の核である「二重支払い否定」を「一度の支払いで」が直接表現する。
- **シンプルさ原則**: 3 案中で最短かつ語順が自然。AI が主語に立つことで「AI が行き渡る」という後半（CEO 承認済み）と主語が連続する。
- **陳腐化耐性**: 特定の課金形態（サブスク・買い切り等）や技術名に依存せず、5-10 年スパンで意味が保たれる。
- **擬人化・物化回避**: 「払った AI」「買った AI」のような違和感を排除。
- **CEO 承認済み後半の保持**: 「開発のすべての工程に行き渡る世界」は変更せず、前半のみ推敲。
