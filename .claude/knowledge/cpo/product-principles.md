# プロダクト原則

MVV.md から導出される、CPO の判断基準。
プロダクト仕様は `docs/specification.md` を参照すること。

## コア原則

<!-- プロジェクトに合わせてプロダクト原則を記述する -->
<!-- 例: ユーザー体験の基準、機能追加の判断軸、UX の優先順位等 -->

1. **MVV 最優先**: 全て MVV に照らす。迷ったら MVV に戻る
2. **ユーザー体験第一**: 技術的な都合より UX を優先する
3. **一貫性**: 過去の判断と矛盾しない。矛盾する場合は理由を明示して更新する
4. **シンプルさ**: 機能を増やすより、既存の体験を磨く方を優先する

## 判断の優先順位

矛盾する要求がある場合の優先順:

1. ユーザー体験の質
2. 操作のシンプルさ
3. 機能の豊富さ

## MVV 設計方法論（2026-05-08 CEO ⇄ CPO 合議より獲得）

MVV を策定・推敲する際に立ち返る判断軸。

### Mission / Vision の書き方

**状態描写型に振る。手段・方法の記述は避ける。**

- NG: 「追加サブスク負担ゼロで〜を成立させる」（手段に縛られ陳腐化する）
- OK: 「追加課金が要らない世界をつくる」（状態を語り 5〜10 年スパンで意味を保つ）

Mission は「何を実現したいか（状態）」、Vision は「実現した世界の姿（状態）」を語る。

### Values の書き方

**動詞起点・2 部構造（対象+動作 または する+しない対比）で揃える。**

- NG: 「観る、伝える、持たない」（3 動詞並列。目的語が消えて意味不明になりやすい）
- OK: 「観察する、書き換えない」（2 動詞で対象と動作・制約を明示する対比構造）

Values は「迷ったとき立ち返る人格」。判断軸として機能するには、読んだだけで意味がわかることが必須要件。

### カッコよさと意味の明瞭さの優先順位

**意味の明瞭さが絶対優先。カッコよさはその範囲内で追求する。**

カッコよさ ≠ 装飾・メタファー・比喩。カッコよさ = 簡潔さ・力強さ・記憶しやすさ。

初見で意味が伝わらないなら判断軸として機能しない。メタファーや詩的表現でカッコよくしても実用性がなければ却下する。

### コアコンセプトの特定

**vibehawk のコア思想「所有を手放すことで自由になる」が Mission/Vision/Values を貫く軸。**

推敲の際は「このコアコンセプトが各項目に貫かれているか」を確認する軸として使う。

- LLM を所有しない（利用者の契約の内側で動く）
- 状態を所有しない（GitHub の既存リソースに置く）
- PR メタデータを所有しない（観察し、伝えるところで止まる）

### 技術原則と Values の階層分離

**技術的設計判断は Values に書かない。`docs/design-philosophy.md` の「設計判断」セクションに書く。**

- Values: 「迷ったとき立ち返る人格」（行動規律・判断軸）
- 技術原則: 「Mission を成立させる構造」（設計上の MUST / MUST NOT）

「状態を持たない」のような技術的設計判断を Values に書くと、階層が混在して両方が弱くなる（Stripe / Basecamp / GitLab の業界定石に従い分離する）。

### 試案への向き合い方

**試案そのままを反映しない。CEO ⇄ CPO 合議でブラッシュアップする。**

「既存試案に引っ張られすぎている」は重要な警告サイン。試案は出発点であり、合議を通じて磨き込む姿勢を保つ。
