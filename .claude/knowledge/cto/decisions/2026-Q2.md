## 2026-05-24 — Issue #222 / PR #223: APPROVE 時のレビュー本文・inline 指摘抑制

### 対象

`scripts/ci/vibehawk-review/post-bundled-review.sh` における APPROVE 時 payload 抑制ロジックの挿入。

### 設計判断: validation 通過後の後段 payload 上書きパターン（intent/bugfix 重視軸）

既存の validation（`.body` の length > 0 チェック）を変更せず、validation 通過後・POST 直前で payload を jq 上書きする設計を採用した。

- **判断理由**: validation を緩める（APPROVE 時は空 body を許容するよう分岐を追加）と、将来の回帰リスクを抱える。validation の意図（body が空の場合は事前に弾く）と抑制の意図（APPROVE 時は空にしたい）が衝突し、分岐が複雑化する。後段に抑制ロジックを挿入することで既存 validation の意図を変えずに済む。
- **intent/bugfix の重視軸「最小修正」との整合**: 修正範囲を POST 直前の 1 箇所に局所化し、既存コードへの影響を最小限にする。

```bash
# validation 通過後・POST 直前に jq で上書き
if [[ "$DECIDED_EVENT" == "APPROVE" ]]; then
  OVERRIDDEN="$(mktemp "$(dirname "$PAYLOAD")/.payload.XXXXXX")"
  jq '.body="" | .comments=[]' "$PAYLOAD" > "$OVERRIDDEN" && mv "$OVERRIDDEN" "$PAYLOAD"
fi
```

### jq + mv の && 連結パターン（payload 破損防止）

jq の変換失敗時に空ファイルで元の payload を上書きしないよう、`jq ... > "$TMP" && mv "$TMP" "$PAYLOAD"` の `&&` 連結を使う。

- `jq` が失敗すると `$TMP` は空または不完全になる。
- `&&` 連結により `jq` 失敗時は `mv` が実行されず、元 payload が保護される。
- 一時ファイルは対象ファイルと同一ディレクトリに作成する（`mktemp "$(dirname "$PAYLOAD")/.payload.XXXXXX"`）。これにより `mv` が同一ファイルシステム内の原子操作になる。
- 既存 `post-bundled-review.sh` 内の他の payload 操作と同じパターンに統一する（コードベース内一貫性）。

このパターンは `shell.md` の「`sed -i` を使わない（BSD/GNU 互換）」推奨パターン（`tmp + mv`）と同じ原理。jq / sed を問わず「変換 → tmp 書き込み → mv」の形式を統一する。

## 2026-05-14 — PR #113: formatDuration 境界値処理と型チェックテスト

### 対象

`cli/setup.js` の `formatDuration` 関数リファクタリングおよび `tests/test_setup_wizard.sh` のアサーション追加。

### 数値フォーマットにおける toFixed 繰り上げ境界値（from CodeRabbit review on PR #113）

`toFixed(1)` は浮動小数点誤差で `"60.0"` を返す場合がある（例: `59999 / 1000 = 59.999`、四捨五入で `"60.0"`）。この値を `"60.0s"` と表示すると利用者にとって意味が不明瞭になる。

対処パターン: `ms < 60000` ブランチ内であっても `toFixed(1)` の結果が `"60.0"` になる場合は分単位に繰り上げる。

```javascript
const seconds = ms / 1000;
const formatted = seconds.toFixed(1);
if (formatted === "60.0") {
  // 繰り上げ
  return "1m0s";
}
return `${formatted}s`;
```

境界値テストは最低限 `59999` / `59500` / `119999` の 3 値を含めること。`59500` は `59.5s` 表示（正常）、`59999` は `"60.0"` 繰り上げ対象の典型値。

### 戻り値の型チェックをテストに含める（from CodeRabbit review on PR #113）

`--dry-run` フラグを持つ関数の戻り値が boolean かつ `true` であることを検証するアサーションを追加した。関数シグネチャの変更に対する回帰テストとして機能する。

```bash
# 例: meetsDogfoodingTarget の型チェック
result=$(node -e "const r = require('./cli/setup.js'); console.log(typeof r.meetsDogfoodingTarget({argv:['--dry-run']}));")
assert_equal "$result" "boolean" "meetsDogfoodingTarget は boolean を返す"
```

## 2026-05-08 — PR #12: vibehawk MVV 制定

### 対象

`MVV.md` 制定・`docs/design-philosophy.md` 状態管理ポリシー追記・`tests/test_smoke.sh` プレースホルダ検証追加。

### テスト実装パターン（`grep -F ... > /dev/null` による終了コード回避）

`set -euo pipefail` 下でコンテンツ検証を行う場合、`grep` のマッチなし（終了コード 1）がスクリプト全体を即死させる問題を `if grep ... > /dev/null; then` 構文で回避するパターンを採用した。

```bash
if grep -F "（ここに" MVV.md > /dev/null; then
  # マッチあり = プレースホルダ残存 → fail
  remaining=$(grep -nF "（ここに" MVV.md)
  fail "MVV.md にプレースホルダが残っている:"$'\n'"$remaining"
else
  pass "MVV.md にプレースホルダが残っていない"
fi
```

既存 `testing.md` の「前提ファイル不在 → `fail()` + `exit 1`」ルールは「全後続テストが無意味になる」場合に発動するもの。MVV.md のような個別アサーションは `if [[ -f ... ]]` ガードで囲むだけで十分であり、`exit 1` は不要。

### 状態管理ポリシーとゲートスタンプの整合確認

`docs/design-philosophy.md` に追記された「vibehawk は専用データストアを持たない」原則と、`~/.cache/vibecorp/state/<repo-id>/` 配下のゲートスタンプは原則レベルで矛盾しない。ゲートスタンプはワンタイムワークフロー制御（review-ok / sync-ok）であり、セッション間の業務状態を保持するデータストアとは性質が異なるため。

### 設計思想ファイルの編集規約

`docs/design-philosophy.md` の冒頭ガイダンスに従い、vibehawk 固有セクション（状態管理ポリシー）を vibecorp 由来セクション（`## vibecorp とは`）より前に配置した。以降の追記も同じ順序を維持すること。

## 2026-05-08 — Issue #6（PR #16）: specification.md vs design-philosophy.md の責務境界確定

### 対象

plan-review-loop 内での C*O 間レビューを経て確定した、2 ドキュメント間の責務分離ルール。

### 確定した責務境界

| ドキュメント | 記載内容 | 管轄 |
|---|---|---|
| `docs/specification.md` | What の宣言（プロダクト仕様・技術スタック） | CPO / CTO 共同（機能 = CPO、技術選定・アーキテクチャ = CTO） |
| `docs/design-philosophy.md` | Why/How の理由（設計判断の根拠） | CTO |

「状態管理原則」のように **仕様宣言と設計根拠が重なるトピック** は両方に記載することが正しい。`specification.md` には「何をするか」、`design-philosophy.md` には「なぜそうするか」を書く。重複は冗長ではなく、参照先の異なる読者に対する意図的な分離。

### bash サンプル付き実装パターンの取り扱い

`specification.md` にインラインで bash サンプルを記載すること自体は仕様の明確化として機能するが、サンプルコードが増えすぎると仕様書としての可読性が下がる（保守性懸念 = Trivial）。

判断: 仕様を理解するために **必要最小限のサンプル** は `specification.md` に置いてよい。実装の詳細（エラーハンドリングパターン等）は `docs/design-philosophy.md` に分離することを検討する。

### ai-organization.md への配置確定（CEO 意図優先）

C*O 間で判断が割れたケース（CPO: 「ai-organization.md は vibecorp 側の関心に近いので移動推奨」/ CTO: 「CEO の完了条件に従い現状維持」）。

スキル仕様「C*O 間矛盾 → Issue 完了条件優先」に従い、CEO が Issue で指定した配置（`ai-organization.md`）を維持することで確定。

**技術的負債として記録**: 将来、`docs/design-philosophy.md` への移動を検討する。`ai-organization.md` が vibecorp 組織設計の説明に終始する場合、vibehawk プロダクト仕様書（`specification.md`）とのスコープ整合が崩れる可能性がある。Issue を立てて CPO・CTO 合意の上で判断する。

### GitHub Actions shallow clone と force push 検出の干渉

`fetch-depth: 1`（デフォルト）で `git merge-base --is-ancestor` を実行すると、過去コミット SHA が履歴から欠落して常に `false` を返す。これにより force push でないのに force push 扱いとなり、完全再レビューが走る誤動作が発生する。

**対処**: 利用者の workflow で `actions/checkout` に `fetch-depth: 0` を指定する（`specification.md` 注 2 として記載済み）。これはコアロジック側では解決できない。利用者側の設定要件として specification に明示するのが正しいアプローチ。

### base ブランチ動的取得（`GITHUB_BASE_REF` 利用）

`origin/main` をハードコードすると、デフォルトブランチが `main` 以外（`master` / `trunk` 等）のリポジトリでの force push 誤検出が発生する。`GITHUB_BASE_REF` 環境変数（PR イベント時に GitHub Actions が自動設定）から動的取得することで解決。

これは `specification.md` の実装パターンコードに `base_ref="${GITHUB_BASE_REF:-main}"` として記載済み。新規実装時も同パターンを踏襲する。
