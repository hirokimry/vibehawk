# CTO 判断記録 2026-Q2

## 2026-06-13 — Issue #346（PR #348）: 配布 workflow の自己完結化

### 対象

npm 配布される `vibehawk-review.yml` / `vibehawk-chat.yml` が外部リポジトリで file not found 全滅していた問題（参照スクリプト約 37 本が vibehawk 本体にのみ存在、setup は workflow 2 ファイルのみ配布）の構造原因と解決パターン。

### 構造原因: 完全一致 dogfooding の盲点

templates ↔ `.github/workflows` の完全一致 dogfooding は「自リポジトリでは動くが外部で壊れる」を隠す。自リポジトリにはスクリプト実体が常に存在するため、**配布物だけで動くか** は dogfooding では検証されない。配布物（workflow / テンプレート）を変更する際は「外部リポジトリには配布ファイルしか無い」前提で参照ファイルを棚卸しすること。

### 自己完結化の標準パターン（runtime checkout 方式）

1. **外部判定**: `if: hashFiles('<guard file>') == ''`。guard file には自リポジトリにしか存在しないスクリプトを使う（shell 分岐不要で step レベル skip できる）
2. **pin 付き 2nd checkout**: `repository: hirokimry/vibehawk` + `ref: <リリースタグ>` + `path: .vibehawk-runtime` + `persist-credentials: false`。runtime checkout は読み取り専用なので token を `.git/config` に永続化しない（CISO 承認条件）
3. **ランタイム基点の決定**: GitHub 式三項演算 `${{ hashFiles(...) != '' && '.' || '.vibehawk-runtime' }}` を `GITHUB_ENV` に書く（shell `if` 不使用 = `workflow-shell.md` 規約準拠で 1 行に収まる）
4. **全 `run: bash` を `"${VIBEHAWK_RUNTIME}/..."` prefix 化**（自リポジトリでは `.` なので挙動不変）

### プレースホルダ `__VIBEHAWK_REF__` の運用

- 配布時に `cli/install.js` が `v<package.json version>` へ置換する。リリースフローは **bump → tag → publish の順序** でタグ実在を保証する
- **説明コメントにプレースホルダのリテラルを書くと置換に巻き込まれる**。コメントでは「配布時に install.js が置換するプレースホルダ」のように文言で言い換える
- 自リポジトリのコピー（dogfooding）では guard により checkout step が skip されるため、未置換のまま残るのが正しい状態（実タグへ「修正」しないこと）

### スクリプト内ファイル参照の位置非依存化

`PROMPT_FILE` デフォルト・include 解決基点は CWD / git toplevel 相対ではなく **SCRIPT_DIR（`BASH_SOURCE`）基点** にする。外部 checkout 実行時、git toplevel は対象リポジトリ（PR 著者支配下）を指すため、参照先不在で破綻する上にプロンプトへの内容注入面になる。一般則として `.claude/rules/shell.md` に追記済み。

### 技術的負債として記録: テスト regex の 3 箇所複製

`test_workflow_yaml` / `test_severity_emoji` / `test_bundled_review_post_step` が同一 regex を複製しており、本 PR の一括変更で追従漏れポイントになった。regex の共通定義への集約は改善余地あり（別 Issue 候補）。

## 2026-06-12 — PR #345: README 大規模再構成における文書検証パターン

### 対象

`README.md` の全面再構成（vibecorp README の 9 見出し構成へ再編、99 insertions / 70 deletions）。

### 文言保全 grep 検証パターン（ドキュメント大規模再構成時）

ドキュメントの大規模再構成では「情報を削除しない」原則の宣言だけでは不十分。plan-review で security-analyst×3 が全会一致で要求した観点は「セキュリティ・免責・課金の重要フレーズを具体列挙して grep で機械検証する」ことだった。

実装したテスト（タスク 10 方式）の要点:

```bash
# 削除してはならない重要フレーズを grep で存在確認する
REQUIRED_PHRASES=(
  "追加課金が発生する"    # 課金条件の明示
  "免責"                  # 免責条項への参照
  "Private Key"           # セキュリティ上の注意
  "required_approving_review_count"  # 設計意図の核心
)
for phrase in "${REQUIRED_PHRASES[@]}"; do
  if grep -q -e "$phrase" README.md; then
    pass "\"${phrase}\" が保持されている"
  else
    fail "\"${phrase}\" が削除された可能性あり"
  fi
done
```

テストは全件パスした。

**判断**: ドキュメント再構成 PR のレビュー観点に「重要フレーズの grep 検証」を標準化する。特に課金・セキュリティ・免責にかかる文言は `intent/docs` であっても Critical 相当として扱い、機械検証を要求する。

### 内部リンク検証での外部 URL 除外（plan-testing 指摘）

`grep -oE '\]\([^)]+\)'` でリンクを全抽出し `test -f` で存在確認する場合、`https://` 始まりの外部 URL を除外しないと偽陰性（外部 URL を内部パスとして `test -f` 評価し必ず fail）になる。

対処パターン:

```bash
# 内部リンク抽出: https:// 始まりを除外
grep -oE '\]\([^)]+\)' README.md \
  | grep -v 'https://' \
  | sed 's/](//; s/)//'
```

**判断**: `shell.md` の `grep -e` パターン終端明示と同じ文脈。`grep` を使ったリンク抽出ではスキーマプレフィックスでのフィルタリングを必ず実施する。

### 見出し順序の機械検証

大規模再構成では見出し順序を目視のみで確認すると CI 再現性がない。`grep -n '^## '` で期待順序と突き合わせる機械検証が有効。

```bash
# 見出し順序を抽出して期待リストと比較
actual=$(grep -n '^## ' README.md | awk -F: '{print $2}' | sed 's/^ *//')
expected="$(printf '%s\n' \
  "⚡ クイックスタート" \
  "🎁 何ができる？" \
  "✨ 何がユニークか" \
  "🛡️ 安全と課金" \
  ...
)"
```

**判断**: `intent/docs` の再構成 PR では「挙動不変性の確認」（`review-observations.md`）と並列して見出し順序の機械検証を要求する。目視確認のみは CI の required check として機能しない。

## 2026-06-07 — ローカルレビュー CLI (`npx vibehawk review`) 技術設計議論

### 対象

`npx vibehawk review` コマンドの新規追加に向けた技術設計。CEO + CPO との合議に供する CTO 見解。

### 論点別判断

#### 1. Claude の呼び出し方

推奨: (a) ローカルの `claude` CLI を `claude -p` で呼ぶ。

- MVV Value 1「利用者の契約だけで、完結させる」は構造的要件。`ANTHROPIC_API_KEY` 従量課金は MVV に直接抵触するため却下。
- `autonomous-restrictions.md` §課金構造は「`claude -p` で LLM を呼ぶ箇所」を不可領域（自律改善ループからの自動変更禁止）として扱う。今回は CEO 承認済みの人間起点実装のため不可領域制約の適用外。
- 技術的リスク: `claude -p` はヘッドレス実行。未ログイン時に exit non-zero またはエラー出力が得られるので検出可能。

#### 2. diff スコープ引数設計

推奨:
- `vibehawk review` = working tree 全体（staged + unstaged）
- `vibehawk review --staged` = staged のみ（pre-commit hook 用）
- `vibehawk review --base <branch>` = base ブランチとの diff（CI と同一スコープ）

3 オプションとも `git diff` wrapper で実装でき外部依存ゼロ。

#### 3. レビュー基準の再利用

CI プロンプトは `templates/.github/workflows/vibehawk-review.yml` に埋め込まれており現状単独ファイルとして存在しない。CLI と共有するには `templates/review-prompt.md` への先行切り出しが必要。これを実施しないと二重管理になり技術負債化する。

#### 4. 出力フォーマット

- デフォルト: `severity絵文字 file:line — 指摘` のターミナルテキスト
- `--format json` / `--format markdown` オプション追加
- `--fail-on <severity>` で終了コード制御（pre-commit / CI 組み込み用）

デフォルト終了コードを non-zero にするかは CPO に委ねる。

#### 5. 前提・エラー処理

必須チェック: git リポジトリ外検出 / claude 未インストール検出 / 未ログイン検出（→ `npx vibehawk setup-token` 誘導）/ 巨大 diff しきい値（`.vibehawk.yaml` の `size_limits` と共通設定を流用する）。

#### 6. パッケージング・テスト

推奨構造: `cli/review.js` エントリ + `cli/review/diff.js` / `cli/review/prompt.js` / `cli/review/output.js`。`testing.md` に従い `tests/test_review.sh` の追加義務あり。

#### 7. アーキテクチャ

既存 `cli/` の責務（セットアップ補助）と review（レビュー実行）は別責務だが同一 npm パッケージ内での拡張として適切。`cli/review/` サブディレクトリで責務分離。

### CPO / CFO / CISO に委ねる点

| 点 | 担当 |
|---|---|
| デフォルト終了コードの是非（pre-commit 必須化）| CPO |
| `claude -p` ローカル呼び出しの OAuth 利用規約整合 | CISO |
| diff 行数しきい値のデフォルト値（Max クォータ影響）| CPO + CFO |

## 2026-05-24 — Issue #222 / PR #223: APPROVE 時のレビュー本文・inline 指摘抑制

### 対象

`scripts/ci/vibehawk-review/post-bundled-review.sh` における APPROVE 時 payload 抑制ロジックの挿入。

### 設計判断: validation 通過後の後段 payload 上書きパターン（intent/bugfix 重視軸）

既存の validation（`.body` の length > 0 チェック）を変更せず、validation 通過後・POST 直前で payload を jq 上書きする設計を採用した。

- **判断理由**: validation を緩める（APPROVE 時は空 body を許容するよう分岐を追加）と、将来の回帰リスクを抱える。validation の意図（body が空の場合は事前に弾く）と抑制の意図（APPROVE 時は空にしたい）が衝突し、分岐が複雑化する。後段に抑制ロジックを挿入することで既存 validation の意図を変えずに済む。
- **intent/bugfix の重視軸「最小修正」との整合**: 修正範囲を POST 直前の 1 箇所に局所化し、既存コードへの影響を最小限にする。

```bash
# validation 通過後・POST 直前に jq で上書き
if [[ "$DECIDED_EVENT" == "APPROVE" ]]; then
  OVERRIDDEN="$(mktemp "$(dirname "$PAYLOAD")/.payload.XXXXXX")"
  jq '.body="" | .comments=[]' "$PAYLOAD" > "$OVERRIDDEN" && mv "$OVERRIDDEN" "$PAYLOAD"
fi
```

### jq + mv の && 連結パターン（payload 破損防止）

jq の変換失敗時に空ファイルで元の payload を上書きしないよう、`jq ... > "$TMP" && mv "$TMP" "$PAYLOAD"` の `&&` 連結を使う（`$TMP` は汎用プレースホルダー。上の APPROVE 例では `$OVERRIDDEN` がこれに当たる）。

- `jq` が失敗すると `$TMP` は空または不完全になる。
- `&&` 連結により `jq` 失敗時は `mv` が実行されず、元 payload が保護される。
- 一時ファイルは対象ファイルと同一ディレクトリに作成する（`mktemp "$(dirname "$PAYLOAD")/.payload.XXXXXX"`）。これにより `mv` が同一ファイルシステム内の原子操作になる。
- 既存 `post-bundled-review.sh` 内の他の payload 操作と同じパターンに統一する（コードベース内一貫性）。

このパターンは `shell.md` の「`sed -i` を使わない（BSD/GNU 互換）」推奨パターン（`tmp + mv`）と同じ原理。jq / sed を問わず「変換 → tmp 書き込み → mv」の形式を統一する。

## 2026-05-14 — PR #113: formatDuration 境界値処理と型チェックテスト

### 対象

`cli/setup.js` の `formatDuration` 関数リファクタリングおよび `tests/test_setup_wizard.sh` のアサーション追加。

### 数値フォーマットにおける toFixed 繰り上げ境界値（from CodeRabbit review on PR #113）

`toFixed(1)` は浮動小数点誤差で `"60.0"` を返す場合がある（例: `59999 / 1000 = 59.999`、四捨五入で `"60.0"`）。この値を `"60.0s"` と表示すると利用者にとって意味が不明瞭になる。

対処パターン: `ms < 60000` ブランチ内であっても `toFixed(1)` の結果が `"60.0"` になる場合は分単位に繰り上げる。

```javascript
const seconds = ms / 1000;
const formatted = seconds.toFixed(1);
if (formatted === "60.0") {
  // 繰り上げ
  return "1m0s";
}
return `${formatted}s`;
```

境界値テストは最低限 `59999` / `59500` / `119999` の 3 値を含めること。`59500` は `59.5s` 表示（正常）、`59999` は `"60.0"` 繰り上げ対象の典型値。

### 戻り値の型チェックをテストに含める（from CodeRabbit review on PR #113）

`--dry-run` フラグを持つ関数の戻り値が boolean かつ `true` であることを検証するアサーションを追加した。関数シグネチャの変更に対する回帰テストとして機能する。

```bash
# 例: meetsDogfoodingTarget の型チェック
result=$(node -e "const r = require('./cli/setup.js'); console.log(typeof r.meetsDogfoodingTarget({argv:['--dry-run']}));")
assert_equal "$result" "boolean" "meetsDogfoodingTarget は boolean を返す"
```

## 2026-05-08 — PR #12: vibehawk MVV 制定

### 対象

`MVV.md` 制定・`docs/design-philosophy.md` 状態管理ポリシー追記・`tests/test_smoke.sh` プレースホルダ検証追加。

### テスト実装パターン（`grep -F ... > /dev/null` による終了コード回避）

`set -euo pipefail` 下でコンテンツ検証を行う場合、`grep` のマッチなし（終了コード 1）がスクリプト全体を即死させる問題を `if grep ... > /dev/null; then` 構文で回避するパターンを採用した。

```bash
if grep -F "（ここに" MVV.md > /dev/null; then
  # マッチあり = プレースホルダ残存 → fail
  remaining=$(grep -nF "（ここに" MVV.md)
  fail "MVV.md にプレースホルダが残っている:"$'\n'"$remaining"
else
  pass "MVV.md にプレースホルダが残っていない"
fi
```

既存 `testing.md` の「前提ファイル不在 → `fail()` + `exit 1`」ルールは「全後続テストが無意味になる」場合に発動するもの。MVV.md のような個別アサーションは `if [[ -f ... ]]` ガードで囲むだけで十分であり、`exit 1` は不要。

### 状態管理ポリシーとゲートスタンプの整合確認

`docs/design-philosophy.md` に追記された「vibehawk は専用データストアを持たない」原則と、`~/.cache/vibecorp/state/<repo-id>/` 配下のゲートスタンプは原則レベルで矛盾しない。ゲートスタンプはワンタイムワークフロー制御（review-ok / sync-ok）であり、セッション間の業務状態を保持するデータストアとは性質が異なるため。

### 設計思想ファイルの編集規約

`docs/design-philosophy.md` の冒頭ガイダンスに従い、vibehawk 固有セクション（状態管理ポリシー）を vibecorp 由来セクション（`## vibecorp とは`）より前に配置した。以降の追記も同じ順序を維持すること。

## 2026-05-08 — Issue #6（PR #16）: specification.md vs design-philosophy.md の責務境界確定

### 対象

plan-review-loop 内での C*O 間レビューを経て確定した、2 ドキュメント間の責務分離ルール。

### 確定した責務境界

| ドキュメント | 記載内容 | 管轄 |
|---|---|---|
| `docs/specification.md` | What の宣言（プロダクト仕様・技術スタック） | CPO / CTO 共同（機能 = CPO、技術選定・アーキテクチャ = CTO） |
| `docs/design-philosophy.md` | Why/How の理由（設計判断の根拠） | CTO |

「状態管理原則」のように **仕様宣言と設計根拠が重なるトピック** は両方に記載することが正しい。`specification.md` には「何をするか」、`design-philosophy.md` には「なぜそうするか」を書く。重複は冗長ではなく、参照先の異なる読者に対する意図的な分離。

### bash サンプル付き実装パターンの取り扱い

`specification.md` にインラインで bash サンプルを記載すること自体は仕様の明確化として機能するが、サンプルコードが増えすぎると仕様書としての可読性が下がる（保守性懸念 = Trivial）。

判断: 仕様を理解するために **必要最小限のサンプル** は `specification.md` に置いてよい。実装の詳細（エラーハンドリングパターン等）は `docs/design-philosophy.md` に分離することを検討する。

### ai-organization.md への配置確定（CEO 意図優先）

C*O 間で判断が割れたケース（CPO: 「ai-organization.md は vibecorp 側の関心に近いので移動推奨」/ CTO: 「CEO の完了条件に従い現状維持」）。

スキル仕様「C*O 間矛盾 → Issue 完了条件優先」に従い、CEO が Issue で指定した配置（`ai-organization.md`）を維持することで確定。

**技術的負債として記録**: 将来、`docs/design-philosophy.md` への移動を検討する。`ai-organization.md` が vibecorp 組織設計の説明に終始する場合、vibehawk プロダクト仕様書（`specification.md`）とのスコープ整合が崩れる可能性がある。Issue を立てて CPO・CTO 合意の上で判断する。

### GitHub Actions shallow clone と force push 検出の干渉

`fetch-depth: 1`（デフォルト）で `git merge-base --is-ancestor` を実行すると、過去コミット SHA が履歴から欠落して常に `false` を返す。これにより force push でないのに force push 扱いとなり、完全再レビューが走る誤動作が発生する。

**対処**: 利用者の workflow で `actions/checkout` に `fetch-depth: 0` を指定する（`specification.md` 注 2 として記載済み）。これはコアロジック側では解決できない。利用者側の設定要件として specification に明示するのが正しいアプローチ。

### base ブランチ動的取得（`GITHUB_BASE_REF` 利用）

`origin/main` をハードコードすると、デフォルトブランチが `main` 以外（`master` / `trunk` 等）のリポジトリでの force push 誤検出が発生する。`GITHUB_BASE_REF` 環境変数（PR イベント時に GitHub Actions が自動設定）から動的取得することで解決。

これは `specification.md` の実装パターンコードに `base_ref="${GITHUB_BASE_REF:-main}"` として記載済み。新規実装時も同パターンを踏襲する。
