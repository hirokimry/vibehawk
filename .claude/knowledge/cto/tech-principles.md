# 技術判断原則

docs/ の仕様書と MVV.md から導出される、CTO の判断基準。
技術スタック・エンジニアリングポリシーは `docs/specification.md` を参照すること。

## 技術選定の判断基準

<!-- プロジェクトに合わせて優先順位を記述する -->

1. **シンプルさ優先**: 複雑な設計より単純な設計。早すぎる抽象化を避ける
2. **依存は最小限**: 新しい依存の追加は、既存の手段で解決できないか先に検討する
3. **テストを書く**: 変更にはテストが伴う

## アーキテクチャ原則

### 状態管理: GitHub リソースのみ（vibehawk 固有）

vibehawk は専用データストアを持たない。セッション間で保持したい情報は Issue/PR コメントまたはラベルに書き、集計が必要な場合は GitHub API で都度取得する。

- ゲートスタンプ（`~/.cache/vibecorp/state/<repo-id>/`）はワンタイムワークフロー制御用であり、セッション間状態保持とは異なる概念。本原則と矛盾しない
- `docs/design-philosophy.md` の「状態管理ポリシー（vibehawk 固有）」セクションが Source of Truth

### 設計思想ファイルへの追記規約

`docs/design-philosophy.md` の冒頭ガイダンス「vibehawk 固有のカスタマイズはこのファイルへ追記してください」に従い、vibehawk 固有セクションは vibecorp 由来セクション（`## vibecorp とは` 以降）より**前**に配置する。vibehawk のオリジナリティを明示するため。

## 依存パッケージの判断基準

- 直近6ヶ月以内に更新がある
- ライセンスが OSS 互換（MIT/BSD/Apache 2.0 等）
- 既存の依存で代替できないか先に検討する
