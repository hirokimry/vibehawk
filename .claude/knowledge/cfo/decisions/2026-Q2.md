### コスト計算の検証

閾値設計の根拠:
- CodeRabbit 互換閾値（30/80/3000）を採用しているため、利用者の既存感覚と一致する
- `3000超 → サマリのみ` の安全弁は、claude-code-action の単一リクエストコスト爆発を防ぐ合理的な上限
- 閾値はすべて `.vibehawk.yaml` でオーバーライド可能なため、大規模モノレポ利用者が追加コストを意図的に受け入れる選択肢が確保されている

### スケール観点

- 並列 PR 実行コストは利用者 workflow の `concurrency:` に委譲済みのため、ツール側のコスト増分は PR サイズにのみ依存する
- 利用者が払う LLM コストは「利用者が既に結んでいる契約の内側で完結」（Value 1）に整合
- ツール側（vibehawk 開発側）に追加 API コストは発生しない構造

### 前提条件

- claude-code-action の リトライ機構が LLM rate limit を吸収する前提（依存）
- `gh api` のバックオフが GitHub API rate limit を吸収する前提（依存）
- 上記 2 点はツール外の仕様であり、変更時は本閾値設計の再評価が必要

### 根拠

ship #5 時点の知見との重複なし（ship #5 は cost-analysis.md 初期構造確立、ship #6 はその上に PR サイズ閾値詳細を追加）。

## 2026-06-13 — Issue #346 / PR #348 配布 workflow 自己完結化（pin 付き 2nd checkout）の CFO レビュー

### 対象

配布 workflow の自己完結化: タグ pin 付き 2nd checkout でランタイム（prompt 等）を vibehawk リポジトリから取得するアーキテクチャ変更。

### 判断内容

#### 1. pin 付き 2nd checkout ランタイム取得

**判断: コスト中立として承認**

- LLM 経路・OAuth 構成は不変（利用者 Pro/Max 枠完結の MUST 制約に抵触なし）
- 追加コストは checkout 数秒分の Actions minutes のみ（public リポジトリは無料 / private は秒単位課金）
- dogfooding（vibehawk 本体）は guard skip により挙動不変
- CFO レビュー記録節を `docs/cost-analysis.md` に追記済み（PR #348 内）

#### 2. プロセス改善（accounting-analyst 2/3 指摘の採用）

**「新規アーキテクチャ採用時は CFO レビューを通すこと」という MUST の記録が cost-analysis.md に欠落していた** → 採用。今後は新アーキテクチャ PR で `docs/cost-analysis.md` への CFO レビュー記録をセットにする。

### コスト構造の変化点（記録）

- **v0.2.3 で外部リポジトリのクォータ消費が本来動作として開始**: v0.2.2 以前は file not found で全滅（= 消費ゼロ）だったため、見かけの消費増は **バグ修正による正常化** であり予算異常ではない。利用量モニタリング時はこの段差を異常検知と誤認しないこと。
- **タグ pin によるコスト予測可能性の改善**: prompt 取得元がバージョン固定になり、main 更新によるトークン消費の予期せぬ変動が消えた。

### 過去判断との一貫性

2026-05-08 / 2026-06-12 判断（「利用者 LLM コストは Value 1 に整合」「ツール側追加 API コストゼロ」）と一貫。LLM コスト構造は不変で、変化は Actions minutes の微増（数秒）のみ。

### 最終判断

**承認: コスト中立。** あわせてプロセス MUST（新アーキテクチャ PR は cost-analysis.md への CFO レビュー記録をセットにする）を確立。
