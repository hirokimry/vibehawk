# CISO プレイブック: docs 再構成レビュー

`intent/docs` かつ全体的な再構成（セクション階層変更・テーブル化等）を伴う PR のメタレビュー定石。

初出: PR #345 / Issue #344 README.md 再構成（2026-06-12）

## 適用条件

以下をすべて満たす変更に適用する。

- `intent/docs`（挙動不変系）の PR
- 複数セクションにまたがる再構成・リライト（部分追記ではなく構造変更）
- セキュリティ設計の保証文言を含む既存ドキュメント（SECURITY.md・README 等）が対象

## 1. CISO Critical 条件フレーズの grep 保全検証（必須）

「情報の削除なし」という自己申告だけでは不十分。以下の CISO Critical 条件フレーズが再構成後も**機械的（grep）に確認できる**ことをテスト基準に要求する。

vibehawk における現行 5 フレーズ（2026-06-12 時点）:

| # | 保全対象 | 確認ポイント |
|---|---------|-------------|
| 1 | OAuth Token stdout 非出力 | 「stdout に出さない」相当 |
| 2 | 3 secrets 手動登録 | 「3 secrets」「手動登録」の両語 |
| 3 | 経路 2 のみ・経路 1 否定保証 | 「経路 1 を標準経路として認めない」相当 |
| 4 | Anthropic 送信開示 | 「Anthropic に送信する」相当 |
| 5 | ANTHROPIC_API_KEY 中止挙動 | 「実行を中止」相当 + [!NOTE] コールアウト形式の維持 |

プロジェクト固有のフレーズは SECURITY.md の「CISO Critical 条件」セクションを参照して最新化すること。

### 計画ファイルへの追記要求（plan-review 時）

実装計画に以下が含まれていない場合は条件付き承認とし、追記を求める。

1. 「セキュリティ文言保全検証」タスク（grep ベース、上記フレーズ全件）
2. テスト計画に「CISO Critical 条件フレーズが grep で保全されている」基準
3. コールアウト形式（[!NOTE] 等）を維持することの明示

## 2. analyst 分裂時の偽陽性判定フロー

analyst が「配置局所化・文脈欠落」系の指摘を出した場合、以下を確認して偽陽性か否かを判定する。

```text
指摘: 「セクション X に局所化されており、文脈から切り離されている」

確認手順:
1. 再構成前（main/base branch）の該当記述の配置を調べる
2. 再構成後の配置と比較する
3. 「再構成前から同じ配置だった」場合 → 今回の差分で新たに局所化が発生したわけではない
4. 情報が他箇所（SECURITY.md・他セクション）にも存在するか確認する

偽陽性判定条件（3つをすべて満たす場合）:
- 再構成前後で配置が実質的に変化していない（階層が 1 段深くなった程度）
- 内容自体は削除されていない
- 同一情報が他箇所にも独立して存在する（情報消失リスクなし）

→ Minor 相当・`intent/docs` 重視軸での修正必須要件を満たさない → 却下
```

## 3. 攻撃チェーン評価（docs 変更特有）

docs のみの変更でも以下の間接攻撃チェーンを評価する。

```text
セキュリティ記述の言い換え
  → 利用者の誤った導入
  → CISO Critical 条件の形骸化
```

- 実現確率は低い（2-3 ホップ）が、新規利用者への情報開示品質に直接影響する
- 保全フレーズが grep で確認できれば攻撃チェーンは遮断されると判断してよい

## 4. 承認パターン

| 状況 | 判断 |
|------|------|
| 計画段階・保全 grep テストが計画に不在 | 条件付き承認（grep テスト追記を要求） |
| 実装後・保全フレーズ全件が grep で確認済み | 無条件承認 |
| 保全フレーズが 1 件でも消えている | 差し戻し |
