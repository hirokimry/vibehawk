### 知見内容

#### Value 4「公式の道を、迂回せず歩く」のセキュリティ含意

Value 4 は単なるプロダクト方針ではなく、セキュリティ設計の根幹となる。

1. **認証経路の限定**: 公式 OAuth フロー以外を選ばないことは、非公式手段（セッションハイジャック・トークン流用）を使わないセキュリティ姿勢と同一
2. **API 使用の限定**: 公式エンドポイントのみ使用することは、API 仕様外の挙動依存（仕様変更時の突然の破損・セキュリティ挙動の予測不能化）を排除する
3. **審査済み経路の優先**: 公式に案内された経路は GitHub によるセキュリティレビューを経ている。迂回することはその保護を自ら捨てることを意味する

#### 「状態は GitHub に置く」のセキュリティ含意

`docs/design-philosophy.md` に追記された状態管理ポリシーは、以下のセキュリティリスクを**設計レベルで排除**する:

1. **情報漏洩リスクの構造的排除**: 自前 DB を持たないため、vibehawk のインフラ侵害が起きても利用者データは GitHub 側にしか存在しない
2. **認証情報管理リスクの排除**: 利用者の GitHub トークン等を vibehawk 側で保管しないため、トークン漏洩の経路がない
3. **アクセス制御の委譲**: 利用者リポジトリの GitHub 権限設定がそのまま vibehawk の操作権限に適用される。振る舞いが意図から逸脱しない構造

#### 残存する攻撃面（Value 4・非保有設計後も残るリスク）

設計で排除できない攻撃面として以下を特定し、`security-principles.md` に記録:

- GitHub Actions での `pull_request_target` + secrets 参照（Fork PR 攻撃経路）
- AI エージェントへのプロンプトインジェクション（PR 本文・コメント経由）
- ゲートスタンプファイルの同一ユーザープロセスからの偽造（信頼境界内のリスク）
- 依存パッケージの既知 CVE

### 反映先

- `docs/SECURITY.md`: セキュリティ方針・認証認可・個人情報セクションを具体化
- `.claude/knowledge/security/security-principles.md`: 脅威モデル・保護すべき資産を vibehawk 固有の内容で記入

### 最終判断

- 承認: セキュリティリスクなし（MVV・設計ポリシーの内容自体は適切）
- 追記のみ: 知見を SECURITY.md と security-principles.md に反映した
